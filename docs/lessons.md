# Lessons Learned

## 2026-03-26: Multi-terminal output performance optimization and end-to-end stress testing

### 1. Test methods must reflect real user scenarios instead of only system metrics
- **Problem**: The initial evaluation only measured terminal output FPS and `xterm.write` latency, then concluded that performance was fine (`fps=32`, `writeMax=0.4ms`), even though typing still felt visibly laggy to the user.
- **Cause**: The analysis mixed up terminal output performance with user input performance. The real pain point was Prompt input responsiveness while terminals were flooding output, not whether terminal rendering itself was smooth.
- **Lesson**: Performance testing must start from the user experience. First identify which operation feels slow, then measure the latency of that operation. Internal metrics such as FPS or write duration are diagnostic signals, not acceptance criteria.

### 2. Stress-test intensity must match real load instead of stopping at a light sample
- **Problem**: The initial stress test only ran for 30 seconds and 25,000 lines, far below the sustained high-throughput scenario produced by Claude Code in real usage.
- **Cause**: The test optimized for fast feedback instead of representing real load characteristics. Claude Code can produce continuous output across multiple terminals for a long time, which is far heavier than a simple loop.
- **Lesson**: Stress tests should run for at least 10 to 20 minutes and use an infinite loop such as `while ($true)` to simulate sustained output. Short tests only expose startup issues and will miss long-run problems such as memory accumulation, GC pressure, and resource leaks.

### 3. Do not rely on overly indirect or fragile end-to-end test methods
- **Problem**: The attempt to automate everything through `app-state.json`, API calls, and background process tricks repeatedly ran into file locks, process-management issues, and terminal ID formatting problems such as stray `\r`.
- **Cause**: The approach aimed for full automation without considering the current infrastructure limits. The API server does not expose terminal creation or layout switching, so mutating state files directly is brittle.
- **Lesson**: Prefer the most direct path. Let the user perform the UI-only step when necessary, then drive the rest through the API. Do not try to bypass every UI step with hacks when the platform does not support it cleanly.

### 4. When editing `useEffect`, always evaluate cleanup impact on initialization flow
- **Problem**: After adding a `setVisibility` effect, the cleanup ran when `visibleTerminals` changed from 1 to 6 terminals, set the old terminals to `visible=false`, and disposed the WebGL context. The next effect immediately recreated WebGL with `visible=true`, and that dispose-plus-create cycle within one frame froze the 6-panel initialization.
- **Cause**: The implementation did not fully account for React `useEffect` cleanup timing. When dependencies change, cleanup runs before the new effect, and that ordering is dangerous for cleanup logic that releases resources such as WebGL contexts.
- **Lesson**: Be especially careful with cleanup in effects that touch GPU resources, DOM state, or other side effects. If cleanup releases a resource that the new effect will immediately recreate, the cleanup may be unnecessary and harmful.

### 5. Performance optimization must start at the source of the data flow instead of patching only the tail
- **Problem**: The first optimization only throttled renderer-side writes to 50ms. That reduced writes per second from 18,000 to 96, but user input latency barely improved (`2212ms` versus the original `1900ms`).
- **Cause**: Renderer-side throttling reduced `terminal.write()` frequency, but the main process still emitted IPC messages every 16ms to every terminal. The renderer main thread still paid the cost of each IPC callback, including map lookups, queue pushes, and scheduling checks.
- **Lesson**: Follow the entire data path from source to sink: `PTY -> TerminalDataBuffer(main) -> IPC -> renderer callback -> pendingData -> rAF -> terminal.write()`. Optimizing only the final write step leaves upstream costs in place. IPC frequency and renderer write frequency both need to be addressed.

### 6. Process management must use exact names and never wildcards
- **Problem**: Using wildcard matches with `taskkill` and `Get-Process` risked terminating unrelated user processes.
- **Cause**: The commands optimized for convenience and ignored the possibility that multiple Onward variants could be running at the same time.
- **Lesson**: This is now a hard rule. Use exact process names only with `taskkill`, `Get-Process`, and `pkill`. On Windows, prefer `wmic process where "name='ExactName.exe'"` for precise querying and control.

### 7. Windows bash variable and PowerShell escaping requires extra care
- **Problem**: When PowerShell commands were sent through `curl`, bash expanded PowerShell variables such as `$i` and `$_` as bash variables, which broke command delivery in most terminals. Python-generated terminal IDs also contained `\r`, which corrupted URL construction.
- **Cause**: Cross-shell invocation chains such as `bash -> curl -> JSON -> PowerShell` introduce multiple escaping layers, and each layer can rewrite or consume special characters.
- **Lesson**: When commands cross shells, prefer heredocs such as `<< 'EOF'` or file-based payloads to avoid multi-layer escaping bugs. Always normalize external command output with `tr -d '\r'` when Windows line endings may be present.

## 2026-04-02: Markdown migration regressions and incomplete first-pass fixes

### 1. Migration review must include shell-level files such as `index.html`, not only component logic
- **Problem**: Markdown images appeared to be wired correctly in the renderer and worker, but the packaged app still failed to display them.
- **Cause**: The migration review focused on `ProjectEditor.tsx` and `markdownPreviewWorker.ts`, but did not compare the source project's `index.html`. The source project allowed Markdown images through CSP with `img-src 'self' data:`, while this repo's initial `index.html` omitted `img-src`, so the browser blocked image rendering even when the generated HTML was correct.
- **Lesson**: When porting a working feature from another project, compare the full execution path end to end: HTML shell and CSP, preload boundaries, renderer state, worker output, and DOM behavior. Do not assume root-level files match the source project unless they have been explicitly diffed.

### 2. End-to-end validation must check visible behavior, not only generated HTML
- **Problem**: The first repair looked successful because the Markdown preview HTML already contained `<img src="data:image/...">`, yet the user still saw broken images.
- **Cause**: The verification standard stopped at internal state and rendered HTML strings. That proved the data path was partly correct, but it did not prove that the browser actually loaded and displayed the image.
- **Lesson**: For UI rendering bugs, autotests must verify the user-visible outcome. In this case the correct assertion is not just "the HTML contains an `<img>` tag" but also "the preview DOM contains images whose `naturalWidth > 0` and whose broken count is zero."

### 3. When a user says the bug still exists, assume layered failures until disproven
- **Problem**: The first Markdown image fix restored the missing re-render after image caching, but the overall issue still remained because a second independent failure was still active.
- **Cause**: The debugging process converged too early on a plausible internal cause and treated that as the whole bug. In reality there were two separate regressions: the missing re-render path and the missing CSP `img-src` allowance.
- **Lesson**: If a user reports that the visible bug still exists after a fix, immediately test for stacked regressions instead of defending the first explanation. Re-check every layer of the pipeline and look for independent blockers that can mask each other.

## 2026-04-02: Settings closing logic must distinguish tab switches from task switches

### 1. Do not apply a tab-switch fix blindly to task switching in the same tab
- **Problem**: After introducing conditional Settings closing, the implementation also closed Settings during `focusTerminal` task switching inside the same tab. In the user flow `Prompt -> manually open Settings -> use task-switch shortcut`, the Settings button and panel disappeared unexpectedly.
- **Cause**: The fix correctly identified that cross-tab navigation needed contextual handling, but it overgeneralized that rule to all shortcut-driven focus changes. Task switching and tab switching are different UX operations and should not share the same closing behavior by default.
- **Lesson**: When a bug report is specifically about cross-tab state, keep the fix scoped to cross-tab transitions unless there is explicit evidence that same-tab task switching should behave the same way. Validate each shortcut path independently.

### 2. Reproduce the exact user interaction sequence before finalizing a state-management fix
- **Problem**: The first revision looked logically correct against the review comments, but it still broke a real user edge case that combined Prompt, manual Settings opening, and task-switch shortcuts.
- **Cause**: The implementation was optimized around inferred state transitions instead of replaying the exact UI sequence the user would perform. That missed a behavior regression in a neighboring path.
- **Lesson**: For UI state fixes, convert the user report into a concrete step-by-step scenario and verify the implementation against that scenario before treating the change as complete. Review comments are useful signals, but they do not replace end-user flow validation.

## 2026-04-05: Git Diff split-view persistence must follow the real drag path

### 1. Do not treat preference writes as proof that the UI interaction works
- **Problem**: The first attempt claimed the Git Diff split width was "remembered" because the code wrote a ratio into `localStorage`, but the user could still drag the sash manually and see no persistence after re-entering Git Diff.
- **Cause**: The validation path only proved that a stored value existed. It did not prove that the manual sash drag path updated that value, nor that reopening Git Diff restored the same visual ratio.
- **Lesson**: For UI persistence bugs, test the exact user path end to end: drag the real control, leave the page, reopen it, and compare the restored layout against the stored value. A stored preference alone is not an acceptance signal.

### 2. Prefer editor layout APIs over DOM guessing for third-party UI state
- **Problem**: Split-width measurement sometimes returned obviously wrong values such as `5px / 5px`, which made the stored ratio meaningless and caused false confidence during debugging.
- **Cause**: The implementation relied on DOM structure and private editor elements that were unstable across render states. Hidden or not-yet-laid-out editors could still satisfy the selector but report garbage geometry.
- **Lesson**: When reading layout state from Monaco or similar editors, prefer public APIs such as `getLayoutInfo()` first, and use DOM probing only as a fallback. The persistence pipeline is only as good as the measurement source.

### 3. Remove experimental self-healing logic before trusting packaged validation
- **Problem**: The intermediate implementation mixed real persistence with simulated drag-on-mount behavior and dangling references from earlier experiments. That made the code harder to reason about and increased the chance of renderer regressions.
- **Cause**: The debugging flow tried to "force" the UI into the desired state instead of first fixing the measurement and persistence chain. Experimental scaffolding remained in the packaged app path for too long.
- **Lesson**: When a UI fix starts accumulating simulated interactions or recovery hacks, stop and simplify. Fix the state source, delete experimental code paths, then rebuild and rerun the packaged end-to-end suite before reporting success.

## 2026-04-08: Subpage navigation UI needs structural analysis before patching visuals

### 1. Visual inconsistency in a shared header usually signals a structural split, not a missing CSS tweak
- **Problem**: The first rounds kept adjusting individual button styles for `Diff`, `Editor`, `History`, and `ESC`, but the user still immediately saw mismatches. The `Save` button in `Editor` exposed that the top area was not truly unified.
- **Cause**: The implementation only shared some CSS classes while each page still assembled its own header actions. That left multiple button structures and style sources alive at the same time.
- **Lesson**: When multiple subpages are supposed to share one top bar, do not stop at "matching classes." Build a real shared header action system with shared button components and shared layout slots. If one button still needs page-local styling, the unification is not finished.

### 2. When the user reports flashing during page switches, inspect the mount lifecycle before proposing cosmetic fixes
- **Problem**: Early fixes reduced some flicker but `Diff -> History` and `Editor -> History` still flashed, because the content area briefly exposed an initialization gap.
- **Cause**: The debugging process focused too much on visible styling and not enough on the actual switch lifecycle. The pages were still responsible for closing themselves before the target page fully opened, which created a body-level empty window during navigation.
- **Lesson**: For flashing or redraw complaints, analyze the DOM and mount order first. Confirm whether the shared shell is stable, whether the body host is reused, and who controls open/close sequencing. Only add transition animation after the lifecycle gap is understood, and use the animation to mask a known initialization window rather than as a blind guess.

### 3. Shared subpage navigation must be orchestrated from one host instead of by each page independently
- **Problem**: Each page initially dispatched the target open event and then immediately closed itself. That made the switching behavior hard to reason about and easy to regress when adding more shared UI.
- **Cause**: Navigation ownership remained distributed across `GitDiffViewer`, `GitHistoryViewer`, and `ProjectEditor`, even after the shell started becoming shared. That meant there was no single place to guarantee "open target first, then retire source."
- **Lesson**: Once multiple subpages share a fixed shell, navigation should be centralized in the host container. Let the host coordinate subpage activation, body transitions, and teardown timing. Do not leave lifecycle ordering scattered across leaf pages.

## 2026-04-25: Git porcelain v2 sub field has four flags — never read just the leading 'S'

### 1. The `<c><m><u>` flags must each be parsed before any submodule UI decision
- **Problem**: Git Diff UI surfaced a parent file-list entry for a submodule whenever the parent's `git status --porcelain=2` reported one, even when the parent's recorded HEAD pointer for that submodule was unchanged. The user then saw "submodule directory was modified" while every file inside the submodule was untouched at the parent level.
- **Cause**: `parseStatusPorcelainV2Z` only inspected `sub.charAt(0)` (`'S'` to detect submodule kind) and discarded `sub.charAt(1..3)` (`<c>`, `<m>`, `<u>`). Any `S.M.` / `S..U` from the parent was treated as a parent-side change, even though those flags describe the submodule's internal worktree, not the parent's index.
- **Lesson**: Porcelain v2's submodule sub field is a 4-character status — `S<c><m><u>`. `c` (commit pointer changed in parent index) is the only flag that is a parent-side change; `m` (work tree dirty) and `u` (untracked content) belong to the submodule's own diff section. Always parse all three flags and gate parent-side surfacing on `commitChanged === true`. `--ignore-submodules` is too coarse — it would also drop legitimate `c=C` events.

### 2. Time-based caches alone cannot deliver "see latest changes after FS mutation"
- **Problem**: A 250 ms request-level Git Diff cache returned stale results when the user modified a file and immediately re-entered the Diff view. Even with the inner fingerprint cache being mtime-aware, the outer time cache short-circuited before the fingerprint check.
- **Cause**: Time-based TTLs answer "how stale am I willing to be" without knowing whether anything actually changed. The fingerprint cache is correct in isolation but is reachable only when the outer cache misses, so the time TTL effectively bounded freshness from above.
- **Lesson**: Real-time-sensitive caches need an *event* invalidation source, not a *time* invalidation source. Pair the cache with a recursive `fs.watch` per active cwd (LRU-bounded so inotify limits stay safe on Linux), debounce raw events, and clear the request-level + fingerprint caches on each window. Keep `force=true` available as a deterministic backstop for subpage entry where the watcher may not yet have observed an FS event before the call lands.

### 3. Subpage entry should always trigger a freshness fetch, not rely on the cache
- **Problem**: Re-entering Git Diff for a cwd already cached returned the cached snapshot, so a user who edited a file in Project Editor and switched to Diff sometimes saw the pre-edit state.
- **Cause**: The mount effect passed `force: !reset`, which meant first-ever opens (the most common entry) did NOT bypass the cache. Re-renders within the panel did. The semantics were inverted relative to user expectation.
- **Lesson**: For UI surfaces whose entire purpose is "show me the current state," always fetch fresh on entry. The watcher-driven cache invalidation is the steady-state freshness mechanism; force-on-entry is the cold-boot guarantee. The two layered together cover both transient race windows (file just changed, watcher hasn't fired) and long sessions (file changed while view was on screen).

### 4. Never static-import `electron` from a module that runs inside a worker_threads context
- **Problem**: After commit `c4625ef` added `import { app } from 'electron'` to `perf-trace-logger.ts`, every git-diff autotest was failing with `Git IPC worker exited with code 1` on the first IPC call. Worker stderr was silent. This silently blocked the staleness regression suite and was the single largest debugging detour in this round — it took aggressive bundle patching (injecting an uncaughtException handler before the require) to surface the underlying `Cannot find module 'electron'` error.
- **Cause**: `git-utils.ts` imports the perf-trace-logger, and `git-utils.ts` is loaded inside the `git-ipc-worker` thread. Electron's worker_threads runtime cannot resolve `require('electron')`; the synchronous require at module-eval time threw before any uncaughtException handler the entry script registered could attach, so the worker died silently.
- **Lesson**: For any module that may be loaded inside a Node Worker thread (anything in `git-utils.ts`'s import graph), `electron` must be resolved lazily via `require('electron')` at call time and gated by `worker_threads.isMainThread`. A static `import { app } from 'electron'` will load fine in main and crash invisibly in the worker. The same rule applies to other Electron-only modules. When debugging silent worker exits, the first diagnostic to add is a `process.on('uncaughtException', …)` handler at the very top of the worker entry, written **before** any other `require()` — and remember TypeScript / esbuild hoist all `import` statements above subsequent statements, so to truly run handlers first you must edit the bundled output or the entry's `require` order directly.

### 5. Per-thread perfTraceLogger singletons must not race on the same `latest.txt`
- **Problem**: After fixing the worker crash, the runner's trace inspection silently picked up the wrong trace file. Main wrote one trace; the worker overwrote `latest.txt` to point at its own (which was missing the main-side `fs-watch-event` records). The GDS-12 assertion failed despite the event firing correctly.
- **Cause**: Each Node Worker has an independent module cache → its own `perfTraceLogger` singleton. Both main and worker called `start()`. The shared trace-root resolver honored `ONWARD_REPO_ROOT` from inherited env, so both wrote into `<repoRoot>/traces/perf/`, racing for `latest.txt`.
- **Lesson**: Process / thread-scoped singletons that touch a shared output directory must hard-pin their output by context. Worker traces should always go to a worker-specific directory (e.g., `${TMPDIR}/onward-traces-perf-worker`) regardless of env vars; only the main thread should write to the project's `traces/perf/`. Test runners that inspect traces should know about both sides and accept events from either trace file.

### 6. Bug-fix work that unblocks a broken test runner WILL surface previously-hidden regressions
- **Problem**: After the worker fix, three additional subpage-navigation assertions (SN-05, SN-06, SN-11) started failing — none of them were in the v0.3 known-failure list. Initial reaction was to suspect the new code; closer read showed the failures were unrelated to my changes.
- **Cause**: Before the fix, every git-diff autotest crashed at the first `git:get-diff` IPC call, so the SN-05/06/11 assertions were never reached. They had likely been red since whichever commit introduced them.
- **Lesson**: When you ship a fix that lets a previously-broken test runner advance further, you will inherit a wave of "new" failures that were always there. Triage them quickly — adopt them into the known-failure list with a clear "newly surfaced after X" note rather than treating them as your bug. The honest accounting ("my fix didn't cause this, but my fix is what surfaced it") protects the next debugger from chasing the wrong cause.

### 7. Subpage-active state must derive from "is target panel still open", not from "panel transition edges"
- **Problem**: The `activeSubpage` sync effect in `TerminalGrid.tsx` was an edge-detector: when a panel transitioned `false → true`, it called `setActiveSubpage(targetId)`. But when the user clicked Diff while autotest's Editor auto-open was still resolving its async cwd lookup, Editor's `false → true` edge fired AFTER Diff's, so the effect clobbered `activeSubpage='diff'` with `'editor'`.
- **Cause**: Edge-detection assumes panels open one at a time. In practice, async auto-open + user clicks routinely overlap. The "latest edge wins" semantics are wrong when the latest edge isn't the user-intended target.
- **Lesson**: Sync effects that mirror state into a derived value should validate that the current value is *still valid* (i.e. its panel is still open) before considering re-derivation. Only sync when the current value is stale. This preserves explicit user intent across racing async opens. The pattern looks like: `if (current is still valid) return; else pick from open panels by priority`.

### 8. SubpageSwitcher button "current" prop must come from the authoritative source, not via a callback round-trip
- **Problem**: SN-05's diff-button-disabled assertion failed because the SubpagePanelShell received `current` from a `panelShellStates[activeSubpage]?.current` lookup. When the destination panel had not yet published its panel-shell state via `useLayoutEffect`, the lookup returned the previously-active panel's state (with its own `current='editor'`), so the diff button rendered as not-current.
- **Cause**: The codebase round-tripped a piece of state — "which subpage is current" — through each panel publishing its own `current` field via a callback. The authoritative source (`activeSubpage` in TerminalGrid) was already known synchronously; routing it through a publish-subscribe sequence introduced a one-frame lag that broke the test's synchronous read.
- **Lesson**: When you have a single source of truth (here, `activeSubpage`), pass it directly to the consumer instead of letting each subordinate component re-derive and publish it. Reserve callback round-trips for state the parent cannot synthesize itself.

### 9. Per-runner `mktemp ONWARD_USER_DATA_DIR` isolation is the FIRST thing to verify when an autotest passes locally and fails in a re-run
- **Problem**: After fixing the SubpageSwitcher race, SN-02/SN-04 started failing on a *re-run* even though they passed on a *first run*. The first run set `tab.activeSubpage='diff'`, persisted that to the user-data dir, and the second run loaded it back via `subpageRestoredRef`, restoring Diff while Editor was also auto-opening — exactly the multi-panel race the new effect was designed to handle, but with the user's intent now ambiguous (was Diff or Editor the target?).
- **Cause**: The runner did not set `ONWARD_USER_DATA_DIR=$(mktemp -d ...)` per the v0.3 hard rule; it shared the global Onward dev user-data dir across runs. State from one test polluted the next.
- **Lesson**: Before chasing app-side behavioral fixes for a "flaky" autotest, run `grep ONWARD_USER_DATA_DIR test/run-<suite>-autotest.sh`. If the runner doesn't `mktemp -d`, that is the bug — not the app. Isolated user-data dirs are how the regression suite stays deterministic; runners that skip it are landmines that turn unrelated PRs into "test broke things" investigations.

### 10. Worker → main perf-trace aggregation: postMessage envelope + per-thread tid lane in main's logger
- **Problem**: After fixing the per-worker trace file race (lesson #5), the next pain point appeared: each worker still wrote its OWN trace file under `${TMPDIR}/onward-traces-perf-worker/`, so a single autotest's events landed across N files (one per worker). Runners had to grep multiple paths, and Perfetto SQL couldn't analyze the workers' work in the same timeline as main.
- **Cause**: Each worker's `perfTraceLogger` was a separate module instance with its own writeStream. There was no aggregation point.
- **Lesson**: For trace systems that span main + workers, the worker should NEVER write its own file. Have the worker's `record()` lazy-detect the worker context and forward via `parentPort.postMessage({event:'trace', name, data, source})`. The main-side worker-client (where the worker was instantiated) listens, replays through main's `perfTraceLogger.record(name, data, {tid: WORKER_TID.X, threadName: 'X-worker'})`. Allocate a stable per-worker `tid` so each worker shows up as its own thread track in Perfetto UI. The pattern was already proven for the ripgrep worker — it just needed to be generalized to the rest of the worker family.

### 11. Process-wide singletons that carry FS watchers belong on the main thread; mark worker-side calls as no-ops with a one-time warning
- **Problem**: After moving the `git-diff-cache-invalidator` `registerWatch` call to the main IPC handler (so the FS watcher could fire IPC notifications to the renderer), the worker's `wireGitDiffCacheInvalidatorOnce` still tried to attach a listener to its OWN copy of the singleton. That listener could never fire (no watcher on the worker side), but it was also a quiet trap — a future refactor calling `registerWatch` from the worker would hit dead code.
- **Cause**: A naive global singleton doesn't know which thread it lives in. Both the main and worker module instances expose the same API surface, but only one of them can functionally register watchers (the main one) because only main can talk to the renderer.
- **Lesson**: For singletons that depend on main-process-only resources (BrowserWindow, ipcMain, the app's IPC channels), gate every public method with `isMainThread`. Worker-side calls become no-ops with a `console.warn` on first invocation per cwd/listener. The warning is the difference between "silently inert" (next debugger spends 30 minutes hunting) and "obviously misconfigured" (next debugger reads the warning and moves on).

### 12. Don't trust the user's literal description as the full bug surface — go look at the actual repo state
- **Problem**: The user reported "submodule appears modified when only the parent was modified" in `/Users/yingyun/Projects/Project_Forward`. I parsed `git status --porcelain=2`'s `<c><m><u>` flags and shipped a fix that handled the three documented scenarios (parent-only, submodule-internal-only, both). The fix was correct *for the scenarios it covered* and passed the synthetic GDS-* fixture. But Project_Forward's actual shape was a fourth scenario I hadn't enumerated: the submodule was declared in `.gitmodules`, the parent commit referenced it, but `git submodule deinit -f` had emptied the working tree and removed the inner `.git` pointer. The directory existed; it was just no longer a git repo. My parent-side filter was correct, but a separate code path in `collectSubmodulesFromGitmodules` was discovering the path from `.gitmodules` and falling back to "treat as a submodule entry" because the only check was `existsSync(subRepoRoot)`. Codex's review caught this in seconds by walking the actual repo state; I had not.
- **Cause**: My loop was: (a) read user's literal phrasing → (b) read porcelain v2 docs → (c) write fix for the documented states → (d) validate on a synthetic fixture I built myself. The synthetic fixture mirrored my mental model of the bug, not Project_Forward's actual shape. Step (d) cannot catch step (a/b/c) blind spots — by design, my fixture validated my understanding, not the world.
- **Lesson**: For bug fixes anchored to a specific real-world repo state, the validation loop must include reading that real-world state, not just synthetic fixtures. Concretely: before designing the fix, run `git status --porcelain=2`, `git submodule status --recursive`, `cat .gitmodules`, and `ls -la modules/<sub>` on the user's reported directory (or ask them to). Then enumerate which states differ from "happy submodule" and write fixture cases for each. The synthetic fixture then covers the OBSERVED state space, not just the documented one. Two minutes of reading the user's repo would have surfaced "this is a deinit-ed submodule" before any code was written. The cost of skipping that step here was a follow-up review round, an extra fixture variant, and a real-world false positive that would have shipped to the user.

### 13. The Git read-side surface needs unification — `getGitRepoMeta`, `parseStatusPorcelainV2Z`, `collectSubmodulesFromGitmodules` each carry partial submodule semantics that drift apart
- **Problem**: The submodule false-positive bug had at least three independent code paths that needed agreement: (a) `parseStatusPorcelainV2Z` decides which entries are submodule rows from `git status` output; (b) `collectSubmodulesFromGitmodules` discovers submodules by parsing `.gitmodules` and `getGitRepoMeta(subRepoRoot)`-validating; (c) `filterMeaninglessSubmoduleEntries` decides whether a parent-side entry is "meaningful" based on `<c><m><u>`. Each of these functions has its own conception of "is this a submodule?" Codex's fix to (b) was orthogonal to my fix to (a) and (c) — each fix patches a different node in the same conceptual graph, with no enforcement that they agree.
- **Cause**: The codebase has accumulated submodule logic incrementally as features (Git Diff, Git History, Project Editor scope, Quick Open) discovered they each needed slightly different views. There is no single `GitRepositorySnapshotService` (or similar) that owns the answer to "given a working tree path, what is the canonical list of (parent, submodule, children, status, sha) tuples?" Each caller cobbles together its own version, and the divergence shows up exactly here — when a corner-case state (deinit-ed submodule) breaks on whichever paths happened not to converge.
- **Lesson**: This is architectural debt, not a bug. The right long-term move is to introduce a unified `GitRepositorySnapshotService` (or equivalent) on the main thread that produces an immutable, fingerprinted snapshot of `{ parent, submodules: [{ path, isInitialized, isRepo, parentSha, headSha, flags }], …}` for any working tree path. All read-side consumers (Diff, History, Editor scope, Quick Open) ask the service rather than re-implementing porcelain-v2 parsing. The snapshot becomes the single place where corner cases like "declared in .gitmodules but not initialized," "initialized but unborn HEAD," "submodule with submodule-of-its-own pointer skew," etc. are enumerated and tested. Until that exists, expect this exact pattern of "fix one path, miss another" every time a new submodule scenario shows up. Track this debt explicitly so the next regression in this area gets the unification done rather than another patchwork fix.

### 14. Phased architectural refactors: ship the service + a compatibility wrapper before migrating individual consumers
- **Problem**: After lesson #13 identified the fragmentation, the temptation was to migrate every read-side consumer (Diff, History, Editor scope, Quick Open) to the new service in one big atomic change. That would have required coordinated updates across ~8 regression suites and risked weeks of ratcheting through unrelated test breakages.
- **Approach taken**: Extract `electron/main/git-repository-snapshot-service.ts` as the canonical answer-owner. Migrate ONLY `loadGitDiff` end-to-end (the highest-value consumer, fully covered by GDS regression). Convert `detectSubmodulesRecursive` into a thin compatibility wrapper that pulls the legacy `GitSubmoduleInfo[]` shape from a snapshot — every other consumer (`getGitHistory`, the `getSubmodules` IPC, etc.) keeps working unchanged because the wrapper preserves the API. Wire the snapshot's own cache into the existing `invalidateGitDiffCache` watcher fan-out so a single FS event invalidates request cache + single-repo cache + snapshot cache in lockstep — there is no second freshness invariant to maintain.
- **Lesson**: For high-blast-radius unification, "phase 1 = new service + behavior-preserving compatibility wrapper" is the right shape. It establishes the architecture, lets the new service grow independent test coverage, and lets later phases migrate one consumer at a time as small, independently verifiable changes. The compatibility wrapper is NOT technical debt as long as it's a shim (zero logic, derives from the canonical answer); it BECOMES debt the moment someone adds branching logic to it. A test like GDS-16 that only asserts "the new service was actually invoked" (via trace event) is enough to gate-keep against regressions where someone accidentally bypasses the service for performance or convenience.

### 15. Dynamic `require()` does NOT survive electron-vite/Rollup output renaming — use static circular imports instead
- **Problem**: The first attempt at the `loadGitDiff` → snapshot service migration used `require('./git-repository-snapshot-service')` inside a function body to sidestep a circular import between `git-utils.ts` and the new module. Source-level typecheck passed; dev build packaged successfully; but the FIRST `git.getDiff` IPC call at runtime crashed with `Cannot find module './git-repository-snapshot-service'`. Every subsequent GDS test failed for the same reason. Required two clean rebuilds + a circular-import audit to recover.
- **Cause**: esbuild content-hashes output chunks (`git-utils.ts` becomes e.g. `git-utils-B-4InHnx.js`). Static `import` statements participate in the dependency graph and the bundler rewrites their resolved paths to match the renamed chunks. Dynamic `require(path)` is opaque to the bundler — the path string is a runtime input, never analyzed, so it gets emitted verbatim. At runtime the resolver tries to find a literal `./git-repository-snapshot-service.js` in `out/main/` and fails because that name no longer exists.
- **Lesson**: For circular dependencies in this codebase's electron-vite + Rollup pipeline, ALWAYS use static `import` statements. ESM live bindings handle circular imports correctly as long as the imports are accessed inside function bodies (not at module-eval time) — both modules finish evaluating before the first call lands, and bindings resolve lazily. Reserve dynamic `require()` only for cases where the path is genuinely runtime-determined (plugin loading, etc.), and verify those work in production builds. `pnpm typecheck` won't catch this — it reads source files, not bundled output. The fast smoke test for a module-resolution issue is "does the dev build's first IPC call succeed?" — these failures explode at first call, never at module-eval, because module-eval-time code paths happen to not reach the bad require.
