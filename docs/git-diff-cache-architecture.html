TL;DRTL;DR

  • 主进程缓存是真正的缓存层（设计意图就是缓存）；渲染端状态不是缓存，是 React state 顺带"持有"了 Monaco 显示所需的字符串。两层概念不同，名字也分开。 Main-process cache is the deliberate caching layer; renderer state isn't a cache — it's the React state that happens to hold the strings Monaco needs to render. Two different concepts, two different names.
  • 每次点击文件，最多触发"渲染端 → 主进程 → Worker → git"四级链路。能在哪一级满足，就停在哪一级，立即把数据沿原路返回。 Each click traces at most a 4-tier ping-pong: renderer → main → worker → git. The first tier that has the answer stops the descent and bounces the data back.
  • 面板上两个独立的 pill 直白告诉你这次点击停在哪一级 — 没有"内容缓存命中"这种把两层混在一起的旧措辞。 Two independent pills in the diagnostics panel say which tier this click stopped at — no more conflated "content cache hit" label.
  • 当前文件变化权威已经不是 Git Diff 自己的 watcher，也不是主进程里的第二套 watcher；而是 GitStateMirror Worker 中的一套 @parcel/watcher，它发出 mirror-update 后再驱动 diff 缓存失效。 The current file-change authority is not a separate Git Diff watcher and not a second main-process watcher. It is the GitStateMirror Worker's @parcel/watcher; its mirror-update drives diff-cache invalidation.

零、当前性校验：旧文档哪里过时了？ 0. Freshness Check: What Was Outdated?

这份文件已合并 cwd / watch / watcher supervisor 流程，并把旧文档里的 watcher 描述更新到当前实现。最关键的变化是：GitStateMirror Worker 是文件系统事件的唯一权威来源，Git Diff 缓存层只消费它发出的 invalidation；watcher 失败时先自动恢复和降级刷新，只有兜底也失败才进入红色 stale banner。 This file now merges the cwd / watch / watcher-supervisor flow and updates the stale watcher descriptions. The key change: GitStateMirror Worker is the single source of truth for file-system events; Git Diff cache only consumes its invalidation signal. 三、VS Code 怎么处理 file watcher 3. How VS Code Handles File Watchers

VS Code 的做法不是“弃用 parcel”，而是“把 parcel 当递归监听快路径，再加一层 supervisor、去重、恢复、降级和诊断”。这对 Onward 的改造方向更有价值。 VS Code does not simply abandon Parcel. It treats Parcel as the recursive fast path and wraps it with supervision, deduplication, recovery, degradation, and diagnostics. That is the more useful direction for Onward.

先纠正一个词： Terminology first: VS Code 这里更准确说是进程模型，不是普通“线程模型”。它把 UI、扩展、主进程服务、文件 watcher 等高成本工作拆到不同进程里，目的是让 watcher burst、native backend 异常、扩展卡顿都不要直接压住 Workbench UI。 VS Code is best described here by its process model, not a simple thread model. UI, extensions, main-process services, and file watchers are split across processes so watcher bursts, native backend failures, and extension stalls do not directly block the Workbench UI.

VS Code 当前策略 VS Code's current strategy

  • 递归目录监听走 ParcelWatcher / parcel-watcher；非递归文件监听走 NodeJSWatcherLibrary / fs.watch。Recursive folder watching uses ParcelWatcher / parcel-watcher; non-recursive file watching uses NodeJSWatcherLibrary / fs.watch.
  • watcher 运行在独立 Utility Process，避免 watcher 计算压力和 native backend 异常直接压住 UI。Watchers run in a separate Utility Process, keeping watcher load and native backend failures away from UI-critical code.
  • watch request 会去重：相同 path / correlation 忽略重复请求；递归 path 重叠时保留最短父路径；文件监听尽量复用已有递归 watcher。Watch requests are deduplicated: identical path/correlation requests are ignored, overlapping recursive requests keep the shortest parent, and file watches try to reuse an existing recursive watcher.
  • 路径不存在或被删除时 watcher 会 suspend / resume；如果没有可复用递归 watcher，就用 fs.watchFile 每 5 秒轮询路径是否回来。If a watched path is missing or gets deleted, watching is suspended/resumed; if no recursive watcher can be reused, fs.watchFile polls every 5 seconds for the path to return.

VS Code 对错误的态度 VS Code's error policy

  • Parcel callback error 会被当作 unexpected error，不假装 watcher 仍健康。Parcel callback errors are treated as unexpected errors; the watcher is not assumed healthy.
  • 对 ENOSPC / inotify limit 这类资源耗尽会特殊识别并只告警一次，避免日志刷屏。Resource exhaustion such as ENOSPC / inotify limits is detected specially and logged sparingly.
  • 普通 unexpected error 会尝试 restart watcher；restart 有延迟，避免磁盘上刚发生的变化导致立刻重启又失败。Other unexpected errors trigger watcher restart attempts; restart is delayed to avoid immediately racing the disk change that caused the restart.
  • 事件量过大时会 throttle / drop，并提示使用 files.watcherExclude 排除高频输出目录。Large event bursts are throttled or dropped, with guidance to use files.watcherExclude for high-churn output folders.
可借鉴但不能照搬： Borrow the pattern, not the whole implementation: Onward 的需求比 VS Code Explorer 窄：我们主要关心 Git 状态是否新鲜。因此更稳的方案可以不是“全量文件 watcher 替换”，而是“parcel 快路径 + Git 状态 polling 降级 + watcher restart supervisor”。 Onward's need is narrower than VS Code Explorer: we mostly care whether Git status is fresh. A more robust design can be “Parcel fast path + Git-status polling degradation + watcher restart supervisor,” not a wholesale watcher replacement.

七、为什么渲染端会"持有"内容？ 7. Why Does Renderer "Hold" Content?

在 Electron 里，Renderer Process 这个名字其实是误导性的 — 它和浏览器里"画像素"的渲染进程不是同一回事。它实际跑了整个 React 应用：组件树、useState、useEffect、事件处理、Monaco 编辑器、所有业务逻辑。 In Electron, "Renderer Process" is a misleading name — it's not the pixel-pushing renderer of the browser sense. It runs the entire React app: component tree, useState, useEffect, event handlers, Monaco editor, all business logic.

你以为 Renderer 在干什么What you think Renderer does 实际在干什么What it actually does
把像素画到屏幕Draws pixels 跑整个 React 应用 — 组件树、useState、useEffect、事件处理Runs the whole React app — component tree, useState, useEffect, event handlers
无状态接收主进程指令Stateless receiver of main-process commands 维护 UI 状态机（哪个文件选中、哪个 hunk 展开、各 panel 折叠状态…）Maintains the UI state machine (selected file, expanded hunks, panel collapse states…)
无内存Stateless React state、ref、事件监听器、Monaco editor models 等大量内存数据React state, refs, event listeners, Monaco editor models — substantial in-memory data

所以 "渲染端持有文件正文" 是 React 应用的结构性后果，不是有意添加的缓存： So "renderer holds file bodies" is a structural consequence of being a React app, not an opt-in cache:

  1. Monaco diff 编辑器需要 OriginalModel + ModifiedModel 两个 JS 字符串模型才能渲染。Monaco's diff editor needs two JS string models to render.
  2. 字符串只能放在 React state — 不能每次重新渲染都从主进程 IPC 拉。Strings have to live in React state — re-rendering can't keep re-firing IPC.
  3. 用户切换文件时，已持有的字符串必须保留，否则切回去要重拉，体验下降。When the user switches files, the held strings must stay in state, else switching back means re-fetching.
  4. fileContents 这个 state 自然就充当了"已经拉过、不必再拉"的指示。The fileContents state naturally serves as a "we already have this" indicator.
类比: Analogy: 说 Word 文档"缓存"了正在编辑的文字 — 听起来很奇怪。它是在持有，不是在缓存。Renderer 的 fileContents 一样。把它叫做"状态"而不是"缓存"，能避免无谓的混淆。 Saying Word "caches" the text you're currently editing — sounds weird. It holds it; that's not "caching." Renderer's fileContents is the same. Calling it "state" rather than "cache" avoids the confusion.

九、一句话总结 9. One-Sentence Summary

主进程缓存是真正的缓存，渲染端状态不是。把这两层用不同的名字、不同的 pill 分开报告 — 整个面板、用户的心智模型、以及你写代码时的边界 — 立刻清楚。 Main-process cache is the real cache; renderer state isn't. Naming them differently and reporting them as two distinct pills clarifies the panel, the user's mental model, and the boundaries you reason about when writing code — all at once.
