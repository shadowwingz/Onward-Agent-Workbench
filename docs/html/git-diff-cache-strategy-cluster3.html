① 概览Overview

一句话:Git Diff 的内容缓存对 staged(index 支撑)条目的新鲜度只锚定工作树 stat,不锚定 .git/index;而请求级缓存对子目录进入用的是子目录键、失效却用 repoRoot 键。两者都让缓存在该刷新时不刷新,返回陈旧内容。 In one line: the Git Diff content cache anchors staged (index-backed) freshness only to the worktree stat, never to .git/index; and the request-level cache keys a subdir entry by the subdir while invalidating by repoRoot. Both leave the cache stale when it should refresh.

🅰 Bug A GDS-22 / GDS-33

staged 文件的 modifiedContent 返回 HEAD/base 内容,而非 index 内容 → 左右两侧相同、看不到 diff。 A staged file's modifiedContent returns the HEAD/base body instead of the index body → original==modified, no diff shown.

🅱 Bug B GDS-15

从子目录进入 Git Diff 后,对仓库根的外部编辑不被监测到 → followup diff 返回空列表。 After entering Git Diff from a subdir, an external edit to the repo root is never picked up → the follow-up diff returns an empty list.
已排除:新增的 git cat-file --batch 串行协议不是 Bug A 的根因——我用最小复现验证了它对 HEAD:f 与 :f 返回的是正确且不同的 blob;readGitFileByRef 也无持久缓存。陈旧内容来自上层 gitDiffContentCache。 Ruled out: the new git cat-file --batch serial protocol is not Bug A's cause — a minimal repro proved it returns correct, distinct blobs for HEAD:f vs :f, and readGitFileByRef holds no persistent cache. The stale body comes from the higher-level gitDiffContentCache.

④ Bug A — staged 内容新鲜度权威content freshness authority

现状策略Current strategy

内容缓存的失效由 FS-watcher(Mirror)驱动的 revalidateContentCacheForProject(git-diff-content-cache-wiring.ts:354)负责:Content-cache invalidation is driven by the FS-watcher (Mirror) via revalidateContentCacheForProject (git-diff-content-cache-wiring.ts:354):

// git-diff-content-cache-wiring.ts
function isWorktreeBackedChangeType(changeType) {           // :154
  return changeType === 'unstaged' || changeType === 'untracked' || changeType === 'conflict'
}                                                       // 'staged' is NOT here

// revalidateProject predicate (:380) — true = EVICT, false = KEEP
(key, staleToken) => {
  if (!isWorktreeBackedChangeType(changeType)) return true   // staged → conservatively evict…
  // …but ONLY runs when a Mirror (worktree) FS event fires
  return current !== staleToken                          // worktree stat changed?
}
缺口:staged 内容由 .git/index 决定,但新鲜度权威是工作树 stat。git add 只改 .git/index、不改工作树文件 mtime,因此 Mirror 的工作树重校验不一定为该 staged 条目触发驱逐 —— 旧的 base body 残留。staged 内容没有任何指向 index 的正向新鲜度锚,只能被动等 Mirror 事件。 The gap: staged content is determined by .git/index, yet the freshness authority is the worktree stat. git add mutates only .git/index, not the worktree file mtime, so the Mirror's worktree revalidation does not reliably evict that staged entry — the old base body lingers. Staged content has no positive freshness anchor to the index; it only waits passively for a Mirror event.

引入于 70d7cfa:把整桶清空换成「按工作树 stat 的 scoped 重校验」,但没为 index 支撑的 staged 条目建立对应的 index 新鲜度权威。Introduced in 70d7cfa: replaced whole-bucket wipe with worktree-stat scoped revalidation, but added no matching index freshness authority for index-backed staged entries.

提案选项 + 权衡Options + trade-offs

方案Option 做法Approach 优点Pros 代价 / 风险Cost / risk
A1 推荐recommended 为 staged 条目引入 index 新鲜度令牌:缓存时记录 .git/index 的 stat 令牌(或暂存 blob oid);scoped 重校验时用「当前 index 令牌 ≠ 缓存令牌」判定 staged 条目是否陈旧。并确保 .git/index 变化能触发 Mirror/失效。Give staged entries an index freshness token: capture a .git/index stat token (or staged blob oid) at cache time; in scoped revalidation evict a staged entry when "current index token ≠ cached token". Ensure a .git/index change triggers Mirror/invalidation. 精确;staged 条目在无关 churn 时存活、仅在真实 index 变化时驱逐 → 保住 EDR 的 spawn 节省。与现有工作树令牌对称。Precise; staged entries survive unrelated churn and evict only on a real index change → keeps the EDR spawn savings. Medium change.
A2 任何 index 变更(应用内暂存/取消暂存 IPC + .git/index watcher 事件)时,显式失效该项目全部 index 支撑(staged)条目。On any index mutation (in-app stage/unstage IPC + .git/index watcher event), explicitly invalidate all index-backed (staged) entries for the project. 简单、正确、易理解。Simple, correct, easy to reason about. 粗粒度:任何 index 变更驱逐全部 staged 条目 → 少量额外重算(staged 条目通常很少,影响小)。Coarse: any index change evicts all staged entries → small extra recompute (staged entries are usually few, so minor).
A3 staged(index 支撑)内容完全不缓存,每次选择都现读。Do not cache staged (index-backed) content at all — read fresh on every select. 最简单;永不陈旧。Simplest; can never go stale. 丢掉 staged 文件的逐文件 body 缓存收益:每次选择都重读 index blob —— 而该缓存初衷正是为 EDR Windows 省掉 spawn 重读。Loses the per-file body cache for staged files: re-reads the index blob on every select — exactly the spawn-taxed reads the cache was built to avoid on EDR Windows.
实现期还需确认一点:无论选哪种方案,我都会在 Plan Mode 前用应用级复现确认「.git/index 变化是否触发 Mirror 事件 / 应用内暂存是否走失效路径」,把精确 file:line 钉死再改。这点不改变上面的策略取舍。 One thing to confirm at implementation time: whichever option you pick, before Plan Mode I will confirm with an app-level repro whether a .git/index change fires a Mirror event / whether in-app staging routes through invalidation, and pin the exact file:line. This does not change the strategy trade-off above.

⑤ Bug B — 子目录请求缓存键subdir request-cache key

这是个更明确的「键作用域」错配,提案唯一且低风险。A clearer "key scope" mismatch; single, low-risk proposal.

现状(陈旧)Current (stale)
// git-utils.ts:1711
getGitDiffRequestKey(cwd) {
  return `${resolve(cwd)}::${scope}`
  // cwd = caller's (a SUBDIR)
}
// invalidation keys by resolved repoRoot
// → subdir key never matches → never cleared
子目录键 …/root/src::full 与失效键 …/root::full 不匹配 → force:false followup 命中陈旧空结果。 Subdir key …/root/src::full never matches the invalidation key …/root::full → the force:false follow-up serves the stale empty result.
提案(修复)Proposed (fixed)
// resolve repoRoot BEFORE keying
const meta = await getGitRepoMeta(cwd)
const root = meta.repoRoot || cwd
const key = getGitDiffRequestKey(root)
// subdir + root share ONE repoRoot key
// which the repoRoot-scoped invalidation clears
子目录与根级调用共用同一 repoRoot 键(getGitDiff 本就解析到 repoRoot 并返回整库 diff),失效正确命中。 Subdir and root calls share one repoRoot key (getGitDiff already resolves to repoRoot and returns the whole-repo diff); invalidation now hits correctly.
权衡:建键前多一次 getGitRepoMeta 解析(loadGitDiff 内部本就在做,且 getGitRepoMeta 自带缓存)→ 风险低,纯恢复正确性。 Trade-off: one extra getGitRepoMeta resolution before keying (loadGitDiff already does it, and getGitRepoMeta is itself cached) → low risk, pure correctness restoration.
