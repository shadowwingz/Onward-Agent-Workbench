文件索引监听后端决策

Project Editor 文件索引（Cmd+P）增量更新失效的根因与后端切换提案

Root cause of the Project-Editor file-index (Cmd+P) incremental-update failure, and a watcher-backend swap proposal

结论摘要Summary

根因（已用数据钉死）Root cause (nailed with data)

在本机当前 macOS（Darwin 25.2.0）上，fs.watch({recursive:true}) 在 宿主 Node v25.2.1 与 App 实际运行的 Electron 39.8.5 / Node 22.22.1 两种运行时下，递归事件都是 0 条。文件索引的增量更新（新建 / 重命名 / 嵌套新建）完全依赖这个监听器，因此 FIC-08 / FIC-11 / FIC-12 / FIC-17 全部失败。这是 产品代码问题（监听后端在新版 macOS 上失效），不是测试用例问题。

On this machine's current macOS (Darwin 25.2.0), fs.watch({recursive:true}) delivers 0 recursive events in both host Node v25.2.1 and the app's actual Electron 39.8.5 / Node 22.22.1 runtime. The file index's incremental updates (create / rename / nested-create) depend entirely on this watcher, so FIC-08 / FIC-11 / FIC-12 / FIC-17 all fail. This is a product-code problem (the watcher backend broke on newer macOS), not a test bug.

提议（还技术债）Proposal (pay back tech debt)

把 project-tree-watch-manager.ts 的监听后端从 Node fs.watch(recursive) 换成 @parcel/watcher——它已经是本仓库的生产依赖（^2.5.6），已经被 git-state-mirror worker 使用并验证可用，也是 VS Code 的递归监听后端。这能确定性修复增量更新，并对齐全仓库的监听技术栈。

Swap project-tree-watch-manager.ts's backend from Node fs.watch(recursive) to @parcel/watcher — which is already a production dependency here (^2.5.6), already used and proven by the git-state-mirror worker, and is VS Code's recursive-watch backend. It deterministically fixes incremental updates and aligns the whole repo's watcher stack.

为什么需要你拍板Why this needs your sign-off

监听器 / 事件过滤语义属于 CLAUDE.md 明确列出的「策略级敏感变更」，必须先可视化「当前 vs 提议 + 权衡」并由你显式决策后才能落地。本文即该可视化；代码尚未编写。

Watcher / event-filtering semantics are explicitly listed in CLAUDE.md as a strategy-level sensitive change that must be visualized (current vs proposed + trade-offs) and explicitly decided by you before implementation. This doc is that visualization; no code has been written yet.

1. 失败现象1. Symptoms

run-file-index-cache-ui 套件中，初始构建相关断言全部通过（FIC-02/03/04），但所有「增量更新后能搜到」的断言失败：

In run-file-index-cache-ui, all initial-build assertions pass (FIC-02/03/04), but every "found after incremental update" assertion fails:

Assertion含义MeaningResult
FIC-08-found-c-after-createIPC 新建 fileC 后能搜到find fileC after IPC createFAIL · results: []
FIC-11-renamed-appears重命名后新名出现renamed name appearsFAIL
FIC-12-original-name-gone重命名后旧名消失old name gone after renameFAIL
FIC-17-nested-propagated嵌套目录新建文件传播到索引nested-dir create propagatesFAIL

关键架构事实：测试通过 window.electronAPI.project.createFile() 这条 IPC 直接建文件，绕过了 ProjectEditor UI。而 UI 自己建文件时会直接调用渲染层的 fileIndexAddFile()；IPC 路径没有这条直连，只能依赖主进程文件监听器把磁盘变更回传渲染层。监听器一旦失效，IPC 建的文件就永远进不了索引。

Key architectural fact: the test creates files via the window.electronAPI.project.createFile() IPC directly, bypassing the ProjectEditor UI. The UI's own create path directly calls the renderer's fileIndexAddFile(); the IPC path has no such direct call and depends entirely on the main-process watcher echoing the disk change back. If the watcher is dead, IPC-created files never enter the index.

2.2 历史回归：4 月全绿，6 月才挂

2.2 History: green in April, failing in June

FIC-11 在全部 4 月 27–28 日的回归里都是 PASS，到 6 月（20260615 / 20260617）变 FAIL。这说明不是测试设计错，而是运行环境（macOS / libuv）发生了变化让监听后端失效——典型的随 OS 升级浮现的平台脆弱性。

FIC-11 was PASS across every April 27–28 regression and only flipped to FAIL in June (20260615 / 20260617). The test design didn't change; the runtime (macOS / libuv) did — a textbook platform fragility surfaced by an OS bump.

2.3 这是 Node 官方记录在案的缺陷

2.3 This is a documented upstream Node defect

  • nodejs/node#55592：macOS 14→15(Sequoia) 升级后，递归 fs.watch 测试「几乎每次都失败」，根因在 libuv 的 FSEvents 桥（libuv#3866），官方只能改测试规避，底层未修。
  • nodejs/node#55592: after macOS 14→15 (Sequoia), recursive fs.watch tests fail "almost every run", root-caused to libuv's FSEvents bridge (libuv#3866); the Node team only worked around it in tests — the underlying defect is unfixed.
  • nodejs/node#47058：递归/父目录监听丢失 ~97% 事件。#52601：macOS 上无法知道监听何时真正「武装」完成，紧随其后的变更被静默漏掉。
  • nodejs/node#47058: recursive/parent-folder watch dropped ~97% of events. #52601: on macOS you can't know when the watch is "armed", so changes right after are silently missed.

4. 提议策略：@parcel/watcher4. Proposed: @parcel/watcher

落地范围（保持语义不变的前提下换后端）：

Scope (swap the backend, preserve the semantics):

  • start() 用 parcelWatcher.subscribe(fullPath, cb, { ignore }) 取代 watch(...)；订阅是异步的，需保存返回的 subscription 以便 stop() 时 unsubscribe()（git-state-mirror worker 已有完整范式）。
  • start() uses parcelWatcher.subscribe(fullPath, cb, { ignore }) instead of watch(...); subscribe is async, so keep the returned subscription and unsubscribe() in stop() (the git-state-mirror worker already has the full pattern).
  • 完整保留：180ms 防抖、忽略过滤（.git/node_modules/.cache/.DS_Store）、目录递归 walk、resync 回退、MAX_PENDING 上限、全部 perf-trace 事件。
  • Fully preserved: 180ms debounce, ignore filter (.git/node_modules/.cache/.DS_Store), recursive dir walk, resync fallback, MAX_PENDING cap, all perf-trace events.
  • parcel 事件自带 type: 'create'|'update'|'delete'，删除事件不再需要 stat→ENOENT 推断，分类更准（修一类历史 race）。
  • parcel events carry type: 'create'|'update'|'delete', so deletes no longer need stat→ENOENT inference — more accurate classification (fixes a class of historical race).
  • 三平台原生后端：macOS FSEvents / Linux inotify / Windows ReadDirectoryChangesW——比 Node 递归 fs.watch 的平台覆盖更稳。
  • Native per-OS backends: macOS FSEvents / Linux inotify / Windows ReadDirectoryChangesW — more robust coverage than Node recursive fs.watch.

5. 同类产品对照5. Peer products

VS Code（最接近的 Electron 编辑器）刻意把递归监听从 Node fs.watch 迁到 @parcel/watcher，并放进独立 UtilityProcess（File Watcher Internals，#132483）；只有 非递归 单路径才用 fs.watch。理由：一个原生库统一三平台，不再维护多套方案。

VS Code (the closest Electron editor) deliberately moved recursive watching from Node fs.watch to @parcel/watcher in a dedicated UtilityProcess (File Watcher Internals, #132483); only non-recursive single-path watches use fs.watch. Rationale: one native library across all three platforms, no more juggling watcher solutions.

@parcel/watcher 是事实标准，被 VS Code、Parcel、Tailwind、Nx、Nuxt 采用。本仓库的 git-state-mirror worker 已经在用它——所以这次切换是对齐，不是引入新风险。

@parcel/watcher is the de-facto standard, used by VS Code, Parcel, Tailwind, Nx, Nuxt. This repo's git-state-mirror worker already uses it — so this swap is alignment, not new risk.

7. 决策选项7. Decision options

推荐 / Recommended  A · 切换到 @parcel/watcherSwap to @parcel/watcher

把 project-tree-watch-manager.ts 的后端换成 @parcel/watcher，语义全保留。根因修复，对齐全仓库与 VS Code，确定性修好 4 个断言并恢复真实外部变更传播。配套：补一个纯逻辑单测锁定事件→{added,removed} 映射，更新 infra/trace.md。符合「还技术债」的本意。

Swap the backend to @parcel/watcher, preserving all semantics. Root-cause fix, aligns repo + VS Code, deterministically fixes the 4 assertions and restores real external-change propagation. Plus: a pure-logic unit test locking the event→{added,removed} mapping; update infra/trace.md. Matches the "pay back tech debt" intent.

B · 仅给 IPC 变更加直连通知（打补丁）Direct-notify from IPC mutations only (patch)

让 createFile/renamePath/deletePath 的主进程 handler 直接广播 PROJECT_TREE_WATCH_EVENT。能让 FIC-08/11/12/17 变绿，但真正的外部变更（外部编辑器、git checkout）仍然进不了索引——监听器照旧坏着，只是测不到了。属于「掩盖」而非「修复」。

Have the main-process createFile/renamePath/deletePath handlers broadcast PROJECT_TREE_WATCH_EVENT directly. Turns FIC-08/11/12/17 green, but genuinely-external changes (external editors, git checkout) still never reach the index — the watcher stays broken, just untested. This masks rather than fixes.

C · 把测试标记为 macOS 平台条件跳过Mark the test macOS-conditional / skip

承认这是宿主平台缺陷，把 4 个断言标记为平台条件性。不改产品，但用户的 Cmd+P 在新版 macOS 上确实漏掉外部新增文件——把已知缺陷藏进绿灯，与「还技术债」相悖。不推荐。

Treat it as a host-platform defect and mark the 4 assertions platform-conditional. No product change, but the user's Cmd+P genuinely misses externally-added files on newer macOS — hiding a known defect behind a green light, contrary to "pay back tech debt". Not recommended.
