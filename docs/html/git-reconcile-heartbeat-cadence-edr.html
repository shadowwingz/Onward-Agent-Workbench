① 结论速览Executive Summary

根因不在预热/聚合，而在 reconcile 心跳。常驻心跳以「固定间隔」(focused 1s / visible 3s) 重复跑 git status，间隔从「上一次 status 完成后」开始计。普通机器 status≈50ms，占空比≈5%，毫无问题。但在本机 EDR 内核 minifilter 下，单次 status 实测 1.3s（中位）~12.9s（峰值），于是「间隔 1s」远小于「status 自身耗时」——心跳几乎背靠背连续地刷 status，把 EDR 的进程创建配额吃满，前台 getDiff 与后台 precompute 抢不到 spawn 槽，于是 Diff 不秒开（91 次缓存查询命中 precompute-pending）。 The root cause is the reconcile heartbeat, not prewarm/aggregation. The always-on heartbeat re-runs git status on a FIXED gap (focused 1s / visible 3s) measured AFTER the previous status completes. On a normal machine status≈50ms → ~5% duty, fine. But under this machine's EDR kernel minifilter, a single status measured 1.3s (median) ~12.9s (peak), so a "1s gap" is far smaller than status's own latency — the heartbeat fires back-to-back continuously, saturating the EDR process-creation budget so the foreground getDiff and background precompute can't get spawn slots → Diff isn't instant (91 cache lookups hit precompute-pending).

The fix is NOT in prewarm — it's in the heartbeat cadence strategy.

③ 根因：节奏假设被 EDR 打破Root Cause: The Cadence Assumption Breaks Under EDR

调度器 git-reconcile-scheduler.ts 把「到期」判定为 now - lastReconcileAt >= 间隔，而 lastReconcileAt 在 status 完成时写入。于是真实周期 = status耗时 + 间隔。它隐含假设 status耗时 ≪ 间隔。EDR 把这个假设反过来了：The scheduler git-reconcile-scheduler.ts deems a repo "due" when now - lastReconcileAt >= interval, and lastReconcileAt is written WHEN status completes. So the real period = statusDuration + interval. It implicitly assumes statusDuration ≪ interval. EDR inverts that assumption:

普通机器（假设成立）Normal machine (assumption holds)

status 50ms + idle 1000ms → 占空比 ~5%，前台随时拿得到 spawn 槽~5% duty; foreground always gets spawn slots

本机 EDR（假设反转）This machine under EDR (assumption inverted)

status 1300~12900ms + idle 1000ms → 占空比 >90%，几乎背靠背；前台 Diff/precompute 抢不到槽>90% duty, nearly back-to-back; foreground Diff/precompute starve

而且 reconcileTick() 对每个到期仓库都是 trackOperation('reconcile', …) 不 await 并发触发：focused 1 个 + visible 多个可同时跑 status，多路并发进一步压垮 minifilter。Worse, reconcileTick() fires trackOperation('reconcile', …) for each due repo without awaiting (concurrently): the focused repo + several visible repos can run status simultaneously, multiplying the load on the minifilter.

④ 当前策略Current Strategy

// git-reconcile-scheduler.ts  (现状 / current)
RECONCILE_FOCUSED_INTERVAL_MS = 1000   // 固定
RECONCILE_VISIBLE_INTERVAL_MS = 3000   // 固定

tick(now):
  for repo in repos:
    if inFlight: skip
    if dirty: due(repo)               // watcher / tab-switch / activate
    elapsed = now - lastReconcileAt   // lastReconcileAt = onReconcileDone 时刻
    if focused && elapsed >= 1000: due(repo, 'heartbeat-focused')
    if visible && elapsed >= 3000: due(repo, 'heartbeat-visible')

特征：①间隔与 status 耗时解耦（不知道 status 多慢）；②无全局并发上限；③无前台让路。在 EDR 下三者叠加 = 持续刷 status。Traits: (1) interval is decoupled from status duration (blind to how slow status is); (2) no global concurrency cap; (3) no foreground yield. Under EDR these compound into continuous status spawning.

⑤ 提案：三个候选策略Proposal: Three Candidate Strategies

A · 自适应退避（按 status 实测耗时）—— 推荐Adaptive backoff (proportional to measured status duration) — Recommended

记录每仓库上次 status 耗时 lastDurationMs，把到期间隔改为 effective = max(baseInterval, lastDurationMs × factor)。占空比被钉死在 1/(1+factor)。Record each repo's last status duration lastDurationMs; change the due interval to effective = max(baseInterval, lastDurationMs × factor). Duty cycle is pinned at 1/(1+factor).

factor = 4:
  status 50ms   → max(1000, 200)   = 1000ms  (普通机器不变 / unchanged)
  status 1300ms → max(1000, 5200)  = 5200ms  (EDR 自动退避 / auto backs off)
  status 12900ms→ max(1000, 51600) = 51600ms (峰值更克制 / spikes restrained)

优 自调优、零配置、普通机器零回归、直击根因；注意 EDR 上 badge 刷新最坏滞后 ≈ status×factor（仍有 watcher 快路径兜底）。Pro self-tuning, zero-config, zero regression on normal machines, hits the root cause; Note worst-case badge lag on EDR ≈ status×factor (watcher fast-path still covers freshness).

B · 固定下限（直接调大常量）Fixed floor (just raise the constants)

focused 1s→5s、visible 3s→10s。focused 1s→5s, visible 3s→10s.

优 改动最小；劣 一刀切：普通机器 badge 平白慢 5s，而 12.9s 的 status 在 5s 下限下仍 ~70% 占空——没真正解决。Pro smallest change; Con one-size-fits-all: normal machines lag 5s for no reason, and a 12.9s status under a 5s floor is still ~70% duty — doesn't truly fix it.

C · 前台让路（Diff/precompute 活跃时暂停心跳）Foreground yield (pause heartbeat while Diff/precompute active)

当某仓库正在前台 getDiff 或 precompute 时，暂停其心跳直到结算。While a repo has an active foreground getDiff or precompute, suppress its heartbeat until it settles.

优 最直接服务「秒开」；劣 需 git-ipc-worker 与 git-state-mirror worker 跨 worker 协调，耦合与复杂度高，且不解决「无 Diff 时心跳仍空转占满 CPU」。Pro most directly serves "instant open"; Con needs cross-worker coordination between git-ipc-worker and git-state-mirror worker — high coupling/complexity, and doesn't address idle heartbeat churn when no Diff is open.

kilocode

github.blog / git-scm		 所有指南都指向「更少/更便宜的 spawn」，没有任何"刷得更勤"的建议：①目录排除；②core.fsmonitor=true（git 2.37+ fsmonitor 守护进程）——订阅 OS 文件事件，让 status 只看 delta，大仓 status 实测降到 <1s。All guidance points to "fewer/cheaper spawns," never "poll harder": (1) dir exclusions; (2) core.fsmonitor=true (git 2.37+ fsmonitor daemon) — subscribes to OS file events so status inspects only deltas, dropping large-repo status to <1s.
对本方案的启示：① Prometheus 原则 = 选项 A（间隔由耗时推导）的权威背书；② kilocode 的否决理由（牺牲正常新鲜度）恰好被 max(base, …) 规避——普通机器 status 50ms 时 max(1000, 200)=1000，节奏与现状一致、零回归，只在 status 真慢时才退避；③ 应保留现有 focus/visibility gating（已具备）；④ 真正治本的 EDR 解法是 FSMonitor（让 status 本身 <1s）——作为后续方向，与本次节奏修复正交。 Implications: (1) the Prometheus principle is the authoritative backing for Option A (interval derived from duration); (2) kilocode's rejection reason (sacrificing normal freshness) is exactly sidestepped by max(base, …) — on a normal machine (status 50ms) max(1000, 200)=1000, identical to today, zero regression, backing off only when status is truly slow; (3) keep the existing focus/visibility gating (already present); (4) the real EDR cure is FSMonitor (status itself <1s) — a follow-up direction, orthogonal to this cadence fix.

推荐 A（自适应退避），并显式保留现有 focus/visibility gating + 设固定下限 = 现状 1s/3s。即 effective = max(focused?1000:3000, lastStatusMs × factor)，factor≈4（占空比上限 ~20%）。经同类调研验证：Prometheus「间隔须大于探针耗时」原则为其权威背书；kilocode 否决纯阶梯退避的理由（牺牲正常新鲜度）被 max(base,…) 规避——普通机器零回归。可选再叠加全局并发上限（避免 visible 多仓库齐发压垮 minifilter）。改动收敛在纯调度器 git-reconcile-scheduler.ts，可单测；watcher 快路径继续兜底新鲜度。 Recommend A (adaptive backoff), explicitly keeping the existing focus/visibility gating + a fixed floor = today's 1s/3s. i.e. effective = max(focused?1000:3000, lastStatusMs × factor), factor≈4 (duty-cycle cap ~20%). Peer-validated: Prometheus's "interval > probe duration" principle is the authoritative backing; kilocode's reason for rejecting a step-ladder (sacrificing normal freshness) is sidestepped by max(base,…) — zero regression on normal machines.
