对账心跳 · Task Git 徽标新版刷新策略

把"徽标何时刷新"从 "只押在文件监听 + 出错才轮询" 改成 统一的 dirty → reconcile 流水线 + 与监听并行的常驻心跳， 让 @parcel/watcher 静默失效时徽标仍能在 1–3 秒内自愈。
配套源码设计文稿：docs/git-status-reconcile-design.md。
焦点仓库 1s 心跳 可见仓库 3s 心跳 按 repoRoot 去重 全在 Worker 线程 ✓ 单测 13/13 ✓ autotest 端到端绿
📌 一句话设计（TL;DR）

徽标新鲜度不再 100% 押在 watcher 上。新增一个 常驻、与 watcher 平行、绝不因 watcher 不报错而失效 的对账心跳：把 watcher 事件 / 切标签页 / 点击 task / 周期心跳 统一汇入一个按 repoRoot 去重的 dirty 队列，由 Worker 线程 每 500ms 取出"到点"的仓库重算 git status——焦点仓库每 1s、可见仓库每 3s。watcher 仍是亚秒级快路径；心跳是兜住"静默失效"的地板。心跳还会在"算出变化但 watcher 近期没报过"时打一条 reconcile-found-drift，把静默失效变可观测。

2背景：旧路径与真实根因

徽标配色速查（5 种脏态 + 干净 / 未知）

每个 Task 标题栏的分支胶囊，圆点颜色由 git status 按「改动类别集合」聚合而来（写死配色，与主题无关）。clean / added / modified 三色语义自始未变；deleted（红）与 mixed（蓝）是把过去一律并入紫色 added 的「纯删除」「增删改混合」两类拆分出来的新态。

状态示意触发条件配色
clean/nullmain无任何改动祖母绿 #10B981
addedmain只有新增类：未跟踪 / 暂存新增 / 重命名 / 拷贝紫 #A855F7
deletedmain只有删除红 #EF4444（= --danger）
modifiedmain只有对已跟踪内容的修改（含合并冲突）黄 #EAB308

分类规则收敛于单一可信源 electron/main/git-status-classify.ts，三套 porcelain 解析器（Mirror Worker / 旧 git-utils RPC / 独立 git-status worker）共用：逐文件归入 {add, del, mod} 之一（重命名/拷贝→add、冲突→mod、删除→del），再按类别集合大小判定——空→clean、单类→对应色、≥2 类→mixed。纯函数由 test/unittest/git-status-classify.test.mts 穷举锁定，端到端配色由 run-git-state-mirror-latency-autotest.sh（GSM-05 mixed、GSM-09c deleted）验证。

真实根因（来自诊断包实锤，非推测）

🎯 根因：@parcel/watcher 静默失效

生产诊断包（v2.1.0-daily.20260530.1）+ 对运行中实例的实时探针证明：在 50 分钟 + 故意建的 6 个文件里，watcher 一条 watcher-fire 都没发、也不报错；唯一的一次重算是 focus-resync。一个 docs/*.md 未跟踪文件徽标停绿 ~5 小时。这正是 parcel-bundler/watcher#187 那类"订阅成功、无错误、却不投递任何事件"的静默失效。

致命点：旧的降级轮询是 "watcher 报错才启动"。watcher 静默失效时它永远不会启动 —— 所有被动监听全失效，徽标只能靠用户恰好 focus 那个终端才更新。（早先怀疑的"忽略名单"/".gitignore" 已被排除：出问题的文件在 docs/，既不被忽略也不被 gitignore。）

🧭 设计原则（与业界一致）

Git 自己的 fsmonitor 把监听当作"需要被校验的缓存，而不是事实源"，丢事件时回退全量重扫。本方案据此把 watcher 降级为"快路径提示"，再叠一层不依赖 watcher 报错的常驻对账作为事实保证。

按 repoRoot 去重队列（约束 H2）

一仓库一周期多个 task 指向同一仓库 → 合并成一次 git status；节奏取最快的（有焦点 task → 1s）。 绝不背靠背onReconcileDone 记 lastReconcileAt，到点前不再 due —— 结构上杜绝"刚算完又立刻算"。 in-flight 排除同一仓库正在算 → 跳过；不重叠。 节奏边界最短 1s（焦点），最长 3s（可见非焦点），隐藏=0（仅显式 dirty 唤醒）。

6三条硬约束

H1 · 性能：轮询与 git status 全在 Worker 线程，绝不在主线程

心跳定时器 + git status 都在 GitStateMirror worker 里跑（与既有 watcher、computeMirrorState、降级轮询同处）。主进程 Router 纯转发：只中转渲染端的焦点信号、把 delta 扇回，不做 git、不跑 reconcile 定时器。符合 Off-Renderer 线程化设计。

H2 · key=repoRoot 的去重队列：一仓库一周期一次 git status

见上节。最短 1s、最长 3s，结构上不背靠背。

H3 · 事件驱动（watcher）全部原样保留并行

parcel-watcher 订阅、classifyEventPath 过滤、80ms 去抖、故障恢复全不动。心跳纯增量——watcher 工作时它几乎不发额外更新（fingerprint 短路）。

8纯调度器内部决策

调度器是零依赖、不读时钟（now 注入）的纯模块——便于单测确定性验证、且可原样搬进 worker bundle。每个 repo 维护 {dirty, inFlight, lastReconcileAt}，task→repo 聚合可见性。

// git-reconcile-scheduler.ts —— tick() 决策（简化）
tick(now) {
  for (const [repo, state] of repos) {
    if (state.inFlight) continue            // 正在算 → 跳过
    if (state.dirty)
      due.push({ repo, reason: state.dirtyReason })  // 显式 dirty：watcher/tab/activate，立即
    else if (vis === 'focused' && now - state.lastReconcileAt >= 1000)
      due.push({ repo, reason: 'heartbeat-focused' })   // 焦点 1s
    else if (vis === 'visible'  && now - state.lastReconcileAt >= 3000)
      due.push({ repo, reason: 'heartbeat-visible' })   // 可见 3s
    // hidden：无心跳，仅 markDirty 唤醒
  }
  return due  // 每 repo 至多一条（去重）
}

可见性聚合：focused > visible > hidden —— 一个仓库只要有任一 focused task 就走 1s。算完调用 onReconcileStart/onReconcileDone 维护 inFlight + lastReconcileAt。这套"数学"由 test/unittest/git-reconcile-scheduler.test.mts 的 13 个断言锁定。

9漂移检测：把静默失效变可观测

这次踩的最大坑是"静默失效不可见"。新设计专门补一条诊断面包屑：

打点watcher 回调每次保留事件时记 lastWatcherFireAt[repo] = now。 判定心跳重算产生了真实变化，但该仓库 now - lastWatcherFireAt > 3s（watcher 近期没报）→ 判定 watcher 漏报。 发射worker:git-state-mirror.reconcile-found-drift（tagged repoRoot/reason/sinceWatcherFireMs）。 价值未来再有"徽标陈旧"上报，trace 一看就知道是 watcher 坏了、不是徽标坏了——本是这次只能靠反复探针才查出的盲区。

10新旧设计对比

✗ 旧：watcher-only + 出错才轮询

  • 新鲜度 100% 押在 watcher 投递事件 + 出错会报错
  • watcher 静默失效(不报错不投递) → 降级轮询永不启动
  • 徽标只能靠 focus-resync 偶发刷新 → 可陈旧数小时
  • 静默失效不可观测（无任何 trace 痕迹）

✓ 新：watcher 快路径 + 常驻对账心跳

  • watcher 仍是亚秒级快路径（H3 保留）
  • 心跳常驻、与 watcher 健康无关：焦点≤1s / 可见≤3s 自愈
  • 按 repo 去重 + fingerprint 短路 → 无变化几乎零成本
  • 漂移检测把静默失效变可观测
  • 全在 Worker 线程（H1），不拖累主/渲染输入

11与既有 Git Diff 缓存 / 流水线的关系

有关系，而且是设计上的强协同——不是孤立的两套东西。关键：GitStateMirror 的 mirror-update（delta）是整条 Git Diff 多级缓存的唯一失效信号源，而对账心跳正是给这个信号新增的产出来源。所以心跳修好的不只是徽标——它顺带把整条 Git Diff 缓存链的新鲜度也一起修好了。

🔗 唯一权威信号：mirror-update

历史上是"5 个 watcher / 11 个缓存"各自盯文件系统；重构后 Mirror Worker 是唯一的 FS 事件权威。gitDiffCacheInvalidator（git-diff-cache-invalidator.ts）不再自己开 watcher，退化成一条纯粹的"失效原因总线"（reason：mirror / watcher-error / force / manual / lru）。Router 一旦收到 worker 的 mirror-update，就调 gitDiffCacheInvalidator.invalidate(cwd, 'mirror')（git-state-mirror-router.ts:336）——一处触发、扇出到下面所有缓存。

完整触发线路：一条 mirror-update 同时驱动「徽标」+「Git Diff 缓存」

这就是"接入"的全貌——不管触发源是 watcher、心跳、切目录还是焦点，最终都汇成同一条 mirror-update；Router 把它一分为二扇出：左路更新徽标颜色，右路失效并重热整条 Git Diff 缓存链。两路共用一个信号源，这就是"零额外管线"的含义。

⚠️ 为什么不会"心跳每秒洗缓存"

fingerprint 短路是这套协同的命门。心跳只在状态真的变了才 emit mirror-update；无变化的 1s/3s tick → 不 emit → 不失效任何缓存、不触发任何 precompute。所以常驻心跳对缓存是友好的——只在该刷时刷。若反过来让心跳盲目每秒 invalidate，会引发缓存雪崩 + precompute 风暴，得不偿失。

🧩 一句话总结

对账心跳不是另起炉灶的第二套缓存/监听，而是给"Mirror = 唯一 FS 权威"这条既有单源信号补了一个不依赖 watcher 报错的产出来源。它天然处在整条缓存流水线的最上游：心跳让 git status 保鲜 → mirror-update → 失效总线 → 请求缓存 / 快照 / 内容缓存 / 预算流水线全部跟着保鲜。fingerprint 短路保证这一切"只在该刷时刷"。

12为什么进 Git Diff 会慢（尤其 Windows）

你的直觉对：切进 git 仓库后本应在后台把 diff 算好，打开就秒开。代码里也确实有两套预热（渲染端 warmDiffCache + mirror 驱动的 precompute）。但在 Windows 上仍会慢——原因是结构性的，下面从源码拆解。

④ 首文件 getFileContentgit show HEAD:file + 读工作区文件1+/文件中
⑤ 渲染端逐文件构建 Monaco diff 模型—中

一次打开 ≈ N（改动文件）+ K（子模块）+ 3 个 git 进程。这条 diff 路径没有任何 Windows 专门分支——慢完全来自"通用 git spawn 在 Windows 上的固有开销"。

🐢 Windows 慢的结构性原因
spawn 密集打开 = 十几到几十次 git 进程（快照 + superproject + 1+K 次 diff + N 次取文件）。 单次就慢Windows 的 CreateProcess 开销远高于 macOS + Defender 实时扫描每个 git.exe 与它读的文件 + Git-for-Windows 的 MSYS 兼容层。同一次 git diff 常是 macOS 的数倍。 force 重算precompute 用 getDiff(force:true) 绕过列表缓存，每次失效都重 spawn；频繁变动的仓库反复重算。 子模块放大full scope 要对每个子模块各跑一次 diff，递归 git 在 Windows 上尤其慢。
⚠️ "后台预热"为何常常落空（命门）

渲染端 warmDiffCache 有 2 秒去抖，且每次 git 状态变化都重置这个定时器（TerminalGrid.tsx:1280），还只在 diff 面板未开时跑。于是：活跃 / 频繁变动的仓库 → 定时器一直被重置 → 预热永远不触发 → 冷开。而 mirror 驱动的 precompute 虽更快（100ms 去抖），但它本身要跑 getDiff(force) + 逐文件取内容——在 Windows 上同样慢，你开得比预热快就还是冷。所以"按理该算好"和"实际还在算"之间，差的就是这段在 Windows 上被拉长的预热时间。

🧩 与对账心跳的关系

这个慢是既有现象，不是心跳引入的。心跳只在真有变化时才发 mirror-update（watcher 本来也会发），所以它对 invalidate/precompute 的增量负载很小。但下面的改进会同时让心跳 + Windows 都更省。

改进方向（建议）

  • 预热不被无限重置：把渲染端 warmDiffCache 的"2s 去抖 + 每次状态变化重置"改成"切入仓库即预热一次"（首次进入立刻起，后续变化只增量补，不归零）。
  • fingerprint 未变就跳过 force：precompute 在 changeFingerprint 没变时复用列表缓存，不重 spawn getDiff(force)。
  • 长驻 git 进程：用 git cat-file --batch / 复用 git status --porcelain 输出，把 per-file 的 N 次 spawn 降为常数次——这是 Windows 上收益最大的一项。
  • 快照延寿：延长 snapshot 缓存 TTL、减少 force 失效，少跑子模块递归。
  • 用户侧：把仓库目录排除出 Windows Defender 实时扫描，立竿见影。
🔬 诚实声明

以上是源码层的结构性原因。精确瓶颈占比（spawn 总数 vs Defender vs 子模块递归 vs 渲染）需要一份 Windows perf trace 实测才能定量——按数据优先原则，真要优化前应先在 Windows 上 ONWARD_PERF_TRACE=1 抓一份，看 main:git.diff.* / main:git.snapshot.* / main:ipc.git.get-file-content 的耗时分布。

16后续方向 / 技术债

  • watcher 自愈：漂移多次时自动重订阅 parcel-watcher，救活快路径（当前仅打 trace + 对账保正确）。
  • OSC 133 命令完成作为又一 dirty 源，给"命令刚跑完即刷新"的跟手感（当前焦点 1s 已够）。
  • getEventsSince 优化：用 parcel-watcher 的快照-since 廉价补回丢失事件，替代全量 git status。
  • 静默失效根因：dev + CDP 复现确认 FSEvents 为何在打包态静默（公开记录指向库 bug / 资源上限 / 卷类型，非 asar / 硬化运行时）。对账层已对此鲁棒。
  • 后台失焦暂停：窗口失焦/隐藏时暂停心跳省电（当前已只覆盖可见仓库）。
