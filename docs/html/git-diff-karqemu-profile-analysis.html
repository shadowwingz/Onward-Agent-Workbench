执行摘要 · 一句话结论

故事已经从"EDR 建进程税"转向更深的问题。内容读取这一块已被本轮未提交的改动基本治好(cat-file batch 让单文件读取降到 1–2ms;submodule 发现降到 2ms)。真正残留的慢是:打开 Diff 时 两个独立 worker 线程同时各跑一遍 git status --untracked-files=all,在 kar-qemu 这种"超大代码树 × 嵌套 submodule"上单次就要 1.2–12.2 秒,而且 缓存的"暖机"被卡在这条多秒的 mirror status 后面 —— 用户往往在暖机完成前就打开 Diff,于是付了全额冷计算。

4缓存何时建立?(回答你的核心问题)

定论Git Diff 缓存是 后台驱动的 —— 在"终端 git 状态变化 / 首次看到某仓"时就开始暖机,不是等用户进入 Diff 界面才懒加载。进入界面刻意"搭便车"读已暖好的缓存(loadDiff({force:false}))。但是:暖机会和进入赛跑,而在 kar-qemu 上进入经常赢不了。

唯一的渲染端 warm 调用点在 TerminalGrid 的 onTerminalInfo 里,门控 repoRoot && !gitDiffOpen TerminalGrid.tsx:1279–1297:首次看到某仓 立即暖机(leading edge),之后状态变化走 2000ms 防抖。Diff 打开 effect 从不暖机,注释明说"读后台暖好的列表缓存,瞬间出图" GitDiffViewer.tsx:3024–3043。

赛跑:为什么 kar-qemu 上进入会输(代码 + trace 双证)

  1. 暖机无法开始,直到 mirror 的慢 status 产出 repoRoot。cwd 切换后第一条 terminal-info 是全 null 占位(repoRoot=null),过不了 warm 门控;repoRoot 只有在那条慢 git status --porcelain=2 --branch -z --untracked-files=all 跑完后才被填上 git-state-mirror-worker-entry.ts:338–369。所以暖机的起步地板 = mirror status 延迟 = 1.8–12.2s(trace 实测),而不是那个 2s 防抖。
  2. 暖机本身在 kar-qemu 上 ~1660ms,且 full scope 还要把 git status 扇到每个 submodule。
  3. 暖机赢的时候,赛跑大体无害:进入的 getDiff(force:false) 和暖机共享 worker 里同一个在途 request-cache 条目(键 resolve(cwd)::scope),后到者搭车而不是再开一次冷算;而且两者在不同 worker lane(进入高优先 cwd;暖机低优先 cwd::diff-precompute),暖机挡不住进入。暖后 getDiff 1–6ms。
  4. 但只要用户在首次暖机完成前打开 —— kar-qemu 的常态,因为光第一次 status 就好几秒 —— 进入就付全额冷算(冷 span [9342,4585,1907]、root-only 8556ms)。这就是"冷启动空窗"。
你的判断:半对驱动确实是后台/事件驱动(不是只在进入时),但管道在两处不完整:(a) 它在 mirror 的慢 status 产出 repoRoot 之前无法暖机;(b) submodule 文件正文被显式排除在内容预热之外 git-diff-content-cache-wiring.ts:200–201,所以即便列表全暖好,第一次点进 submodule 文件仍是冷读。

5问题 1:点击修改文件应读缓存,不应每次建进程

P1点击文件 = 缓存读取,不建进程基本已缓解
裁决

命中预热文件的点击是纯渲染端内存命中(source:'renderer-memory'),不过 IPC、不建 git 进程。残留 miss 路径确实存在,但即便 miss,Windows 上那次 git 对象读取也走 cat-file batch 管道往返,不是新建进程(trace:batch spawned=2 / fallback=0)。所以"每点一次就建一次 git 进程"在当前构建上并没有发生。

什么时候会 miss 落到 IPC
  • 文件 key 被 watcher/mirror 标记为 stale(staleFileContentKeysRef);
  • 渲染端预取还没轮到它(预取是 250ms 防抖后对整张可见列表串行 for-await,无 top-N 上限 GitDiffViewer.tsx:3795–3876);
  • force 被置位(手动/自动刷新);或 diff 列表 reload 把该条目丢了。
142 次 content miss 的真正来源(不是点击)

主内容缓存在每次 mirror-update 被整桶清空,而渲染端预取又在每次列表身份变化时对整张列表重打一遍 —— 于是清空后窗口里发出的每个 fetch 都是 L2 miss。142 远超"12 文件 × 点击",正是这种"整列表反复预取打在刚清空的桶上"的 churn,而非点击驱动的冷读。

精确修正(对抗式验证)

L2 miss 时 worker 的读取次数因状态而异:unstaged 修改=1 次 git 对象读(原始侧=index blob,:gitPath)+ 1 次工作树 fs 读;staged 修改=2 次 git 对象读(HEAD blob + index blob)。win32 上每次对象读都是 batch 管道往返,从不新建进程。

6问题 2:退出再进入,整个操作重来

P2重进应瞬开部分真实
裁决

同 cwd、期间无失效的重进是 ~1ms 的 request-cache clone(实测重进 1ms,会话内 0 次失效)。重进变冷只在两种情况:(a) cwd 自上次后变了;(b) 退出→重进的间隙里发生了真实失效。在 kar-qemu 真实使用中 (b) 频繁发生,所以你"重进又重来一遍"的体感是真的 —— 但根因是访问间隙里的失效 churn,不是"缺重进缓存"。

一次 mirror-update 的失效扇出(无作用域、无防抖)

一条 worker mirror-update 会对 msg.cwd 和 msg.state.repoRoot 各触发一次失效;经 gitDiffCacheInvalidator 同步点燃 3 个监听器,合力清空 4 层读侧缓存(request 的 root-only+full 两键、single-repo、snapshot、content 桶),再前推清空 git-ipc-worker 自己的副本,再通知渲染端(GIT_DIFF_CACHE_INVALIDATED),并调度一次预热(这一臂有 200ms 防抖,但清缓存与强制 reload 两臂没有)。没有任何按文件的作用域。git-state-mirror-router.ts:335–337 · git-utils.ts:460–492 · content-cache-wiring.ts:269–309

间隙里真实会触发失效的源(均已确认)
  • 用户自己在终端跑的 git 命令(checkout/commit/add)—— 它们的 .git/{HEAD,index,refs,logs} 改动是 classifyEventPath 允许放行的;
  • 在 Project Editor 里保存一个被跟踪文件;hunk 暂存/取消暂存/丢弃;
  • 指纹是 mtimeNs:size:mode(已排除 ctime,所以纯 NTFS ctime churn 不再翻转 —— 这是本轮已修的点),但真实编辑/改大小仍翻转指纹 → 清空所有读侧层。
更大的隐藏成本当面板开着时遇到失效,渲染端会发 loadDiff({force:true}) 强制全量 reload(外加强制重取选中文件)GitDiffViewer.tsx:3122–3124。force:true 绕过 L5/L6 付全额冷算([9342,4585,1907]),这很可能就是 trace 里三段 ~60s 渲染端 event-loop stall 的来源 —— 比你描述的"关闭后重进"更伤。

7问题 3:submodule 工程首次打开特别慢

P3submodule 首开慢真实 · 当前主要矛盾
裁决

结构发现已不是问题(fs 化,2ms 冷)。慢的是每仓 git status --untracked-files=all 的固有成本,被 (父 + 每个 submodule) 相乘,又被两个 worker 各跑一遍,在后台暖机来得及之前以全额冷价付掉,Windows 上再叠加每进程 EDR 税。

成本公式

full scope 的 getDiff 对父仓 + 每个 submodule 各跑一次 getSingleRepoDiff(Promise.allSettled 扇出 git-utils.ts:2077–2094),每个发一次 git status --porcelain=2 -z --find-renames=50 --untracked-files=all + 最多两次 git diff --numstat。即固有成本 ≈ (1 status + 0..2 numstat) × (1 + N submodule)。其中 --untracked-files=all 的工作树遍历是主导项,对 OS 缓存极敏感 —— 三次同样的 force 衰减 9342 → 4585 → 1907 正是缓存升温(进程数不变)。

为什么 kar-qemu 冲到 12.2s 而 Onward-Github 只到 4.2s

mirror 的超级工程 status 没带 --ignore-submodules git-state-mirror-worker-entry.ts:338–349,于是 git 递归进每个 submodule,-uall 把嵌套 node_modules 里的未跟踪内容也枚举一遍。watcher 的 ignore globs 只压制 FS 事件,压不住 git 自己的 status 遍历 —— 两个独立的面。

修正冷延迟由"未跟踪遍历"与"~40 次 spawn 的 EDR 税"共同主导,两者都随 OS FS 缓存升温;trace 衰减与此一致,但不能单独把遍历认定为唯一主因。"每次 spawn 数相同"是从代码路径推断,非直接观测。

9根因总览(按严重度排序)

Critical两个 worker 线程冗余各跑一遍 git status(无共享 status 缓存)

GitStateMirror worker 与 git-ipc worker 是两个独立线程、各有独立缓存。一次 kar-qemu 开 Diff,两者都对同一超级工程跑 git status --porcelain=2 --untracked-files=all(mirror 在 attach/switch;ipc-worker 在 getSingleRepoDiff),同一命令类、同一解析器、零复用。在单次 status 就 1.2–12.2s 的仓上,这把最贵的操作翻倍。worker-entry.ts:338–369 · git-utils.ts:1912–1922

Criticalgit 固有成本 × submodule 树 + 未跟踪遍历 + EDR 建进程税

full getDiff 是 (1 status + 0..2 numstat) × (1 + N submodule);mirror 因没带 --ignore-submodules 又额外递归进 submodule。--untracked-files=all 工作树遍历是主导且对 OS 缓存极敏感(冷衰减 9342→4585→1907)。Windows 上未被 batch 覆盖的 status/numstat/rev-parse spawn 各付 ~1–3s EDR 税。并发已拉满,唯一的杠杆是每仓更少/更便宜的命令,不是更多并行。

High全有或全无的失效扇出:无作用域、关键两臂无防抖

一次 mirror-update 经 3 个监听器无作用域地清空 4 层读侧缓存(+ worker 副本 + 渲染端通知),对 cwd 和 repoRoot 各来一遍。预热臂有 200ms 防抖,但清缓存与强制 reload 两臂没有。指纹只要翻转就发(已排除 ctime,所以纯 ctime churn 不再触发)。这是 P2(间隙失效后重进变冷)、P1(高 miss 比)、~60s 渲染端 stall 的共同根。

High暖机起步被卡在慢 mirror status 之后(冷启动空窗)

warmDiffCache 在 terminal-info 带上非 null repoRoot 之前无法开始,而 repoRoot 只有那条多秒的 mirror git status 跑完才有。于是暖机起步地板 = mirror status 延迟(kar-qemu 1.8–12.2s),用户在此窗口内打开就付全额冷算。管道是后台驱动的,但在大 submodule 仓上"暖"来得太晚。

Medium双 scope 暖机(root-only + full,分键)

warmDiffCache 同时算 root-only 和 full,落在不同 request-cache 键,列表层互不满足。修正:父仓 status+numstat 不会被双重执行(有确定性的在途去重,非竞态)。真正的代价是两个 scope 在首次冷暖机时各自跑了父仓的结构+内容工作,且 full 还要跑 submodule 扇出 —— trace 的 8.5s root-only + 6.8s full 反映的是这个,而非父仓 status 被重复执行。

Medium开 Diff 时按 submodule 订阅 mirror,放大 attach 重算

GitDiffViewer 为超级工程 + diff 结果里每个 submodule root 各订阅一次 mirror,每个 0→1 refCount 触发一次 attach 全量 git status。修正:超级工程通常不额外触发(终端常驻 mirror 已持 refCount≥1,面板订阅是 no-op);kar-qemu 6 次重算是真的,但未按 reason 拆分,"每 submodule 开 Diff 触发"是"支持但未证明"。

Lowsubmodule 文件正文被排除在内容预热之外

预热 scheduler 的 isEligible 对 isSubmoduleEntry 返回 false content-cache-wiring.ts:200–201,所以 submodule 文件正文永不预热,第一次点进 submodule 文件总是冷读 —— 一个 submodule 专属的 P1 缺口,直接叠加到 P3。

10改进方向(仅分析,本轮不实现)

以下是综合分析得出的去冗余/提速方向,本报告未改动任何代码。按收益排序:

1
在 mirror worker 与 ipc worker 之间共享超级工程的 status
mirror 已算出父仓完整的 porcelain-v2 变更集,getSingleRepoDiff 又从零重算同一条。用一个按 repoRoot + index/mtime token 的主进程 status 缓存,或让 diff 路径直接消费 mirror 已解析的父仓结果,可消掉最贵的那次重复操作。权衡:mirror 用 --branch、diff 用 --find-renames=50,复用会丢父仓重命名检测,是产品取舍。
2
收窄 submodule 的 status 遍历
mirror 超级工程 status 没带 --ignore-submodules 导致递归,而 diff 路径又对每个 submodule 单独 getSingleRepoDiff —— submodule 改动集可能被枚举两遍。给超级工程 status 加 --ignore-submodules=dirty/all 即可去掉递归遍历(很可能就是 12217ms vs 4187ms 差距的来源),前提是终端 dirty 指示能容忍非递归汇总。
3
合并双 scope 暖机:full ⊇ root-only
full 是 root-only 的严格超集(父+submodule vs 仅父)。让 root-only 从 full 结果派生,而不是第二次独立冷算,可消掉 trace 里 ~8.5s 的 root-only span。
4
可推导处去掉冗余 numstat
numstat 是叠在 status 之上的第二次 git diff 调用,虽已按 hasUnstaged/hasStaged 门控,但小改动集可从单次合并 diff/status 推出增删行数,省掉每仓每次最多 2 次 EDR 税 spawn。
5
给失效扇出加作用域 / 防抖
一次 mirror-update 无作用域地清掉两个键全部 4 层缓存;纯指纹变化(文件列表没变)仍清整个 content 桶。利用 mirror 已有的每文件指纹,把 content 失效收窄到真正 stat 翻转的文件,可保住大部分 109 次命中,而不是逼出整列表重预取(142 miss 的主因)。
6
别在每次列表身份变化时重预取整张可见列表
渲染端预取无 top-N 上限、每次列表身份变就重跑,于是每次失效后 reload 都对刚清空的 L2 打一轮整列表预取(miss=142 的结构放大器)。把预取限定到视口窗口,或新旧列表 diff 后只取变化项,可砍掉冗余 IPC 量而不影响可见文件的点击延迟。
7
cwd == repoRoot(或 cwd 是子目录)时去掉重复失效
router 对 new Set([msg.cwd, msg.state.repoRoot]) 失效;当终端 cwd 是子目录时会发两次 invalidate → 两次渲染端广播 → 两次开面板强制 reload。收敛到规范 repoRoot 可在该情形减半广播/reload。

11结论

已经治好的(本轮未提交改动的成效,实测确认):文件内容读取 1.3s → 1–2ms(cat-file batch);submodule 结构发现 8–10s → 2ms(fs 化);进程数 ~69 → 40;纯 NTFS ctime churn 不再触发失效;暖缓存命中时进入/重进/点击都是 ~1ms。

仍然慢的本质:不再是"建进程",而是 git status 本身在 kar-qemu 上就要好几秒,并被三件事放大 ——(1)两个 worker 各算一遍(Critical);(2)mirror status 递归进 submodule(Critical);(3)暖机被卡在这条慢 status 之后,用户能赢过赛跑就吃冷算(High)。再叠加无作用域的失效扇出(High)让真实使用中缓存反复被清空。

建议的下一步优先级(若后续动手):先做改进方向 #1(共享 status)+ #2(收窄 submodule 遍历) —— 这两条直接打掉两个 Critical,把 kar-qemu 冷开的主导成本砍掉一大半;再做 #5/#6(失效作用域 + 预取收窄) 稳住暖缓存,解决 P1 的 miss churn 与 P2 的间隙失效;#4(warm 起步解耦慢 status) 填上冷启动空窗。
