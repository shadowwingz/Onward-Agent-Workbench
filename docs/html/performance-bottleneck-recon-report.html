Onward 性能优化总体方案 & 数据摸底报告

合订文档:上半部(①–⑦)是按方案 Phase 0「先量测」原则对 7 个用户可感 surface + 性能基础设施做的一次数据摸底(硬证据回答「哪块最可能是瓶颈」);下半部(Part B · §0–§12)是完整的性能优化 & 性能基础设施总体方案(现状图谱 + 缺口矩阵 + 分阶段路线图 + 三块 greenfield 基建设计)。摸底正是本方案 Phase 0 的产物,二者互为「量测 ↔ 蓝图」。

① 核心结论(TL;DR)

一句话:已优化、有门禁的「warm/输入」路径不是瓶颈;真正的瓶颈集中在「冷开 / open 路径 / renderer 线程同步解析」,而它们恰恰是当前埋点为空、量不到真实数字的盲区。

① Prompt 输入是全场唯一「测得到 + 有硬门禁」的金标准 surface——p95/p99/max 均有签署预算并在每次回归中卡构建。它几乎可以从瓶颈嫌疑中排除。

② 最高风险的两块是 Git Diff 与 Git History 的 open / 详情渲染路径:GitDiffViewer.tsx 7,139 行 / 191 个 hooks,open 时跑「无上限、串行、整列」预取;GitHistoryViewer 的 parsePatchFiles + @pierre/diffs 同步跑在 renderer 线程的非 memo 函数里、每次重渲染重算、且无结果缓存。两者的主进程 IPC 路径都零埋点,现在无法量到真实耗时。

③ File-Open / Monaco:大文件(≤30MB)整串过 IPC,且 waitForEditorModelReady 每 16ms 做一次整文件字符串比较轮询直到模型就绪——三重 O(文件大小) 成本叠加,无任何 open 延迟门禁。

④ Quick Open / 冷启动:文件名模糊搜索每个按键触发一次 IPC(无 debounce),冷缓存下还会走 queue.shift() 的 O(n²) 目录遍历;应用冷启动零埋点。

⑤ 基础设施三大空白:内存监控 = 0、CPU 采样 profiler = 0、trace 回归门禁虽接线但已死(从不设 ONWARD_PERF_COMPARE_BASELINE、无任何已提交 baseline)。用户上报的 trace 是本仓库唯一诊断输入,但最贵的 Git IPC 路径在 trace 里没有任何面包屑。

对方案的一处事实更正:方案 §1.1 称「~449 events」——实测 src/utils/perf-trace-names.ts 注册的事件数是 179,449 是该文件的行数。本报告后续按 179 计。

② 方法与口径

两层数据,互为补充,且都拒绝「凭代码猜瓶颈」。

  • 层 A · 静态硬指标(9 个只读 agent 并行扇出):对每个 surface 用 grep -c / wc -l / 读源码,产出带 file:line 证据的精确计数——组件规模、hook 数、同步阻塞调用点、算法复杂度、埋点死/活、缓存层清单。这些是瓶颈风险的代理量,可在不构建的情况下得到。
  • 层 B · 真实延迟实测(构建 dev 包后跑 6 个已有的延迟 autotest):harvest 真实 p50/p95/p99/max。这是「先量测」的金标准数据,但只覆盖那些已经埋了点的 warm 路径。
为什么大多数瓶颈「现在量不到真实数」? 方案点名的 open/冷路径(MAIN_IPC_GIT_GET_DIFF、MAIN_IPC_GIT_GET_HISTORY、tab-switch duration、冷启动 span 等)在 perf-trace-names.ts 里已注册但从未 emit——即便现在驱动 app 去开 Git Diff,trace 里也是空白。所以这些只能先给静态风险评级,真实数字要等 Phase 0 把这些 span 接活后才有。本报告用「数据可得性」标签如实区分三态。

延迟实测的聚合与防抖口径

沿用仓库既有的 N-trial 规则:布尔正确性 → N 次全过;延迟预算 → N 次中至少 1 次达标才算过(承认 GC/调度瞬时尖峰,只在系统性回归时失败);吞吐 → 取中位数/p95。各 runner 的预算常量见第④节。

PART B · 方案

Onward 性能优化 & 性能基础设施 — 总体方案

状态:PROPOSAL — 2026-06-01。这是一份 plan-first 文档:方案评审通过前不改任何生产代码,通过后逐阶段执行,每阶段由用户把关。本方案由一次只读「领地测绘」(13 个并行读码 agent,覆盖 7 个用户面 + 6 个基建维度)+ 一次同行研究(4 个 agent 研究 VS Code / Chromium / Electron / Node,引用见 §12)产出——每条「现状」均出自读真实源码,每条架构建议都有同行先例支撑。上半部①–⑦的数据摸底正是本方案 Phase 0「先量测」的产物。

启动 Q&A 锁定的 4 项决策:

  1. 预算先量测(Measure-first)。~15ms「无感」理想只适用于输入与warm 缓存交互。冷开(Git Diff/History、文件打开、冷启动)物理上做不到 15ms——涉及 git 子进程 / 磁盘读 / Monaco 挂载。Phase 0 给每个面埋点、跑 N 次、产出真实延迟分布,交给用户,然后由用户设定每面预算并编码为门禁。
  2. 混合运行姿态(Hybrid)。内存 + CPU 基建:到处廉价常采样;重型捕获(堆快照 / CPU profile)仅在阈值 / 按需 / 调试构建触发,带单发 + 冷却 + OOM 螺旋保护。
  3. CPU profiler = V8 inspector .cpuprofile 便携基线(全进程、跨平台、无需 OS 权限)+ OS 原生深挖层(macOS sample/spindump,Windows WPR/ETW,Linux perf)作为隔离的 per-OS 适配器,按需触发。
  4. 先门禁后优化(Gate-before-optimize)。一个面在同时具备功能 autotest 与延迟门禁之前不优化。

§3 Phase 0 — 验收门禁设计(「先量测」)

整个倡议的命门:没有门禁无法证明优化,没有先量测无法设定公平预算。Phase 0 不挑预算——它产出让用户能挑预算的数据。

3.1 每面两层门禁

  • 功能 autotest —— 重构热路径时能抓到行为回归。多数面已有(缺口:tab 创建/切换无;旧终端路径无)。
  • 延迟门禁 —— 带显式、产品签署预算的计时断言,用仓库的 N-trial 聚合规则。

3.2 量测循环(每面按序)

  1. 端到端给 open/冷路径埋点。先接活已注册但死的 span(MAIN_IPC_GIT_GET_DIFF、RENDERER_IPC_GIT_GET_DIFF、MAIN_IPC_GIT_GET_HISTORY、RENDERER_MARKDOWN_MERMAID),并补缺的(冷启动 main:app.ready/main:window.created/renderer:first-paint;Quick Open per-query span;tab 切换 duration span;预取突发完成)。
  2. 写延迟 autotest:对真实 fixture 跑真实路径 N 次(N=5 布尔 / N=3 三选一,按仓库计时规则),把分布(p50/p95/p99/min/max)记入日志——初始给宽松占位预算使其不会失败。
  3. 跑它、收割真实数,把每面 per-OS 的 p50/p95/p99 呈给用户。
  4. 用户设预算。编码为具名常量(如 GIT_HISTORY_OPEN_BUDGET_MS = …)带 // signed-off: <date> 注释。
  5. 锁定为 runner 的硬门禁,并(Phase 1)进入 trace 回归基线。

3.4 聚合与抗抖

采用同行验证纪律(hyperfine / vscode-perf-bot,§12):跑 N 次,丢弃首次(冷),再聚合 warm 样本。布尔正确性 → 全 N 过;延迟预算 → N 次中至少 1 次达标(承认瞬时 GC/调度尖峰,系统性回归才失败);吞吐 → 中位数/p95。这已是 CLAUDE.md 家规,Phase 0 让它跨面统一。

§4–6 三块 Greenfield 基建

§4 基建 A — trace 输出回归门禁

目标:每次回归运行回读已捕获的 NDJSON,若某跟踪事件的延迟相对已提交基线回归超容差带则失败。
同行依据:Chromium Catapult 用变点/异常检测跑时间序列(非平阈值);Android Benchmark 明确拒绝作者维护的平基线;VS Code vscode-perf-bot 用 best-of-N + 单一绝对常量;TracerBench 提供 Mann-Whitney U / 置信区间原语;hyperfine 预热后丢弃冷跑。

设计 — 纯 Node 跑后门禁(test/autotest/perf-gate.mjs,热路径不依赖 Perfetto):① 流式解析 NDJSON,只留 {ph:'X',dur,name,pid/tid},按 surface:event-name 分桶(µs→ms 一次性转换);② 抗抖聚合(N=5,丢首次,池化 warm,算 p50/p95/p99+stddev);③ 每事件双门:(a) 绝对预算(p95 是否优于产品签署预算,镜像 Constants.FAST)、(b) 基线 delta(候选分布是否显著慢于已提交基线;先用 % 容差带——简单、确定性,可选 @tracerbench/stats 置信区间升级,需存原始样本数组);④ 已提交基线 perf-baseline.<os>.json 每套件每 OS 一份,刷新走评审提交;⑤ 激活休眠比较器作为过渡;⑥ 加 infra/scripts/Open-Trace.ps1 补 Windows 分析平价。配套单测:resolvePhase、sanitizeArgs/PII 脱敏、enforceBudget 驱逐数学、checkRateLimit 令牌桶、normalizeTraceValue;外加实测开销基准取代假设的 ~5µs/事件。

§5 基建 B — 内存看门狗 + 堆转储(混合)

同行依据:Node 三种触发(--heapsnapshot-signal=SIGUSR2、--heapsnapshot-near-heap-limit=N、程序化 v8.writeHeapSnapshot());快照需 ~2× 活堆、同步(秒级-分钟级)、且自身可能触发它在诊断的 OOM。VS Code 生产环境不自动转储——而是 Process-Explorer + 按需/CI 快照。renderer 快照走 CDP HeapProfiler / webContents.takeHeapSnapshot。关键洞见:renderer RSS 增长常是 Chromium 资源缓存(图/字体/css),在 V8 堆快照里不可见——只有 webFrame.getResourceUsage() 看得到,webFrame.clearCache() 是比转储更便宜的首选缓解。

Tier 1 廉价常采样:main 持有的单一 MemoryWatcher,5–10s 定时扇出三路廉价读:app.getAppMetrics()(每个 Electron 进程)、main 线程 v8.getHeapStatistics()(used/limit 比 + number_of_detached_contexts 强泄漏信号)、对每个 worker 线程的 IPC ping 自报(worker 对 getAppMetrics 不可见,必须自报)、每个 renderer 报 webFrame.getResourceUsage()。全部作 ph='i' 事件发射,用户构建默认 OFF(ONWARD_MEM_WATCH=1 启用)。
Tier 2 重型堆转储,三重门控:阈值(滑窗 + used/limit>0.85 且持续上升)、按需(env 单发 + 调试 IPC/菜单)、安全网(--heapsnapshot-near-heap-limit=2)。OOM 螺旋护栏(不可妥协):全局单飞互斥 + per-pid 冷却 + 会话上限 + 需 N MB 余量。跨平台:V8+process.memoryUsage 三 OS 一致;residentSet 在 macOS 缺失(用 private);--heapsnapshot-signal 在 Windows 失效(按需走 IPC/菜单)。

检测器 = 纯函数(可单测):{timeseries} → {pressure: none|warn|critical},滑窗取最近 ~12 个样本(≈2min),仅当 used/limit>0.85 且持续上升/正斜率才触发。产物生命周期:traces/heap/(dev)/ userData/debug/(prod),命名 Heap-<ts>-<pid>-<thread>.heapsnapshot,按大小上限淘汰最旧(单份 50–500MB),标记为含 PII(活字符串 = 文件路径 / 缓冲文件内容 / env 值)——绝不自动上传,仅用户显式操作才进诊断包。renderer 转储走 webContents.takeHeapSnapshot,main/worker 走 v8.writeHeapSnapshot(各 isolate 转储自己,main 无法转储 worker 堆)。诊断面包屑:main:memory.pressure-detected、main:memory.dump-written、main:memory.dump-skipped(附原因:cooldown / in-flight / insufficient-headroom)。

§6 基建 C — CPU profiler(V8 基线 + OS 原生深挖)

同行依据:VS Code 用 v8-inspect-profiler(CDP Profiler over inspector port)。Node inspector.Session → Profiler.* → .cpuprofile;--cpu-prof 用于启动期。renderer 经 webContents.debugger(与开着的 DevTools 互斥)。挂死的主线程无法 profile 自己 → Chromium base::HangWatcher 是进程外/看守者模式。

Tier 1 便携 V8 基线(跨平台、无 OS 权限):main 一个 ProfilerService,IPC profiler:start/stop,目标由 app.getAppMetrics() 驱动。main:new inspector.Session() → 写 traces/cpu/main-<ts>.cpuprofile;每个 worker:在其内部建自己的 inspector.Session().connect()(非 connectToMainThread);renderer:webContents.debugger.attach,守 DevTools 互斥。

§7 renderer 卸载、暖启动 & 缓存治理器

同行依据:Monaco 把语言服务 + diff 计算放 Web Worker(WorkerBasedDocumentDiffProvider);VS Code 把词法化留主线程但增量/让步/可见优先。DOMPurify 需要真实 DOM——把它移进 worker(linkedom/jsdom)会静默欠净化(mXSS),是有记录的安全隐患(cure53/DOMPurify#577、GitLab MR!74963)。React 调度器用 MessageChannel + 5ms 切片 + isInputPending;标准化的 scheduler.postTask 暴露 user-blocking/user-visible/background。Node module.enableCompileCache / NODE_COMPILE_CACHE(v22.1+)缩短 worker 冷启。

7.1 移出 renderer 线程的内容(按收益/风险排序)

  1. @pierre Git-History diff 解析 → 池化 worker 线程。纯文本进 / 结构化 diff 出,Monaco 先例,最高收益、最低风险、无 DOM。缺口矩阵里单项最大的 renderer 响应性赢点。
  2. markdown 解析 + highlight.js + KaTeX → worker(string→HTML string)。然后在 renderer 跑真实 DOMPurify,但只对可见视口切片,经 scheduler.postTask 分块,绝不阻塞按键。把成本拆开:O(n) 解析/高亮/数学离开 renderer,安全关键的净化留在可信 DOM。不要把 DOMPurify 移进 worker。
  3. 池化 markdown worker(停止每次切文件 terminate()+重建,经 owner-id 路由复用)并对所有 worker 入口应用 NODE_COMPILE_CACHE。

7.2 暖启动 & 治理器(跨切面)

  • 磁盘持久暖启动 renderer 文件索引 + last-working-set 清单(仅路径,不含正文)到 userData,启动时再水合 + 重校验(mtime/size/hash)。正文永不持久(新鲜度 + PII)。
  • 启动期预判预热 在 MainWorkScheduler 后台道:首输入前主动暖上次项目的文件索引 + diff working-set + git 快照。
  • 中央缓存治理器:对 diff content/request/snapshot/file-index 缓存统一 trim(pressureLevel) 接口,由 Tier-1 内存采样器(§5)喂养——缓存在真实压力下收缩,而非各守固定预算。
  • 把「prompt 输入优先」升为全局不变量:今天该模型只对终端输出执行。让 Git-status 扇出与全局搜索都查 inputPriorityLane.shouldYieldToPromptInput()。

§8 缓存新鲜度张力(§7/子页工作前须解决)

最高杠杆的子页优化——未变更仓库重开时信任 Git Diff/History 缓存而非强制重取——直接抵触 docs/lessons.md 教训 #3(「子页进入应总是触发新鲜抓取,而非依赖缓存」),后者源自一个真实 bug(基于时间的缓存在 FS 变更后显示陈旧状态)。

为何现在重访可能安全:该教训早于 GitStateMirror。Mirror 现在是仓库工作树是否变化的单一权威(它拥有 parcel-watcher + porcelain-v2 状态 + 变更指纹,并扇出 delta)。所以安全表述不是「信任缓存」,而是:「重开时,当且仅当该仓库根的 GitStateMirror 代际自缓存项写入后未推进才信任缓存,否则强刷」。既保住教训 #3 的保证(总能看到变更后状态),又在常见的未变更情形跳过冗余重取。这是对正确性敏感的改动:先带详细设计 + 专门 autotest(驱动教训 #3 的精确失败路径)再动重取路径,不单方面改。
