用长驻 git cat-file --batch 消除「看 Diff 的建进程税」 Eliminating the "Diff View Spawn Tax" with a Long-running git cat-file --batch

把"每点一个文件就 fork 两个 git 进程"换成"一仓一个常驻进程,只走管道"。在受 EDR 限制的 Windows 上,单文件读取 1263ms → 3ms(~420×)。 Turning "spawn two git processes per file click" into "one long-running process per repo, pipe-only". On EDR-throttled Windows, single-file read drops 1263ms → 3ms (~420×).

Onward · Git Diff Performance · Phase A · 2026-06

TL;DR

  • 问题:看 Diff 时每点一个文件,主进程会 fork 两个 git 进程(cat-file -s 取大小 + cat-file blob 取内容)。在装了 EDR/DLP 的公司 Windows 上,内核 minifilter 给每次 CreateProcess 收 ~0.4–3s 的"过路费",于是每点一下卡 ~1.3s。 Problem: every file click forks two git processes (cat-file -s for size + cat-file blob for content). On corporate Windows with EDR/DLP, a kernel minifilter taxes each CreateProcess by ~0.4–3s, so each click stalls ~1.3s.
  • 方案:一仓一个长驻 git cat-file --batch 进程。建进程的成本只付一次,之后每次读取都只是"往 stdin 写一行 ref,从 stdout 读回内容"——不再有新进程,把"建进程税"换成"管道往返"。 Solution: one long-running git cat-file --batch process per repo. Pay the spawn cost once; every later read is just "write a ref to stdin, read content from stdout" — no new process. The spawn tax becomes a pipe round-trip.
  • 跨平台:机制平台无关,只有"用哪个 git 命令"按平台解析。Windows 已实现;macOS 留了 placeholder(将来接入打包的平台特有 git);Linux 不启用(原生建进程本就便宜,走原 per-call 回退)。 Cross-platform: the mechanism is platform-agnostic; only "which git command" is resolved per platform. Windows done; macOS has a placeholder (for a future bundled command); Linux not enabled (native spawns are cheap there — falls back to per-call).
  • 绝不弄坏 Diff:双层兜底 —— 平台门控 + 运行时 catch 降级到老的 per-call 路径。最坏只是退回"慢但正确"。 Never breaks the diff: two-layer safety — platform gate + a runtime catch that degrades to the old per-call path. Worst case is "slow but correct".

1问题:为什么 Windows 上看 Diff 不"秒开"The problem: why Diff isn't "instant" on Windows

Onward 把 git 工作放在 git-ipc-worker 里跑。看 Diff 时,渲染端每选一个文件,就要拿到它的"原始内容"(index 里的 blob)和"当前内容"。读 blob 的老实现走 readGitFileByRef,对每个 ref 依次 fork 两个 git 子进程: Onward runs git work inside git-ipc-worker. In Diff view, each file the renderer selects needs its "original" (the index blob) and "current" content. The old blob read in readGitFileByRef forked two git children per ref: 

# 老路径:每读一个 blob 就两次 CreateProcess / per blob = two CreateProcess
git cat-file -s    <ref>   # 取大小 / size
git cat-file blob  <ref>   # 取内容 / content
Root cause 公司 Windows 装了 EDR/DLP(内核 minifilter,如 aTrust / IsaAgent / Fp*·Ws* 驱动,并非 Windows Defender —— D:\ 与 Git 已加白)。它在每次 CreateProcess 时同步检查,给每个 git 进程加 ~0.4–3 秒的固定开销。于是"读 1 个文件 = 2 次 spawn ≈ 2.6s",多点几下就是肉眼可见的卡。 Corporate Windows ships an EDR/DLP kernel minifilter (e.g. aTrust / IsaAgent / Fp*·Ws* drivers — not Windows Defender; D:\ and Git are already excluded). It inspects every CreateProcess synchronously, adding a fixed ~0.4–3s per git process. So "read one file = 2 spawns ≈ 2.6s", and a few clicks become visible lag.

3方案:一仓一个长驻 cat-file --batchSolution: one long-running cat-file --batch per repo

Git 自带一个为"批量取对象"设计的长驻协议 cat-file --batch:进程起一次,之后通过 stdin/stdout 持续应答,不再建新进程。我们不重写 git、不引库、不背 GPL —— 只是用用户自己的 git,换一种调用姿势。协议如下: Git ships a long-running protocol designed for bulk object reads — cat-file --batch: start the process once, then keep answering over stdin/stdout with no new process. We don't reimplement git, add a dep, or take on GPL — we just drive the user's own git differently. The protocol: 

# 起一次:一仓一个常驻进程 / start once, one per repo
git -c core.quotepath=false cat-file --batch

# 之后每次读取 / each read thereafter:
stdin :   <ref>\n
stdout:   <oid> <type> <size>\n   # header
          <size bytes of content>\n  # body + trailing LF
# 缺失对象 / missing object:
stdout:   <ref> missing\n

每个常驻进程把请求串行喂进去(响应按输入顺序从单一 stdout 回来),用一个状态机解析 header 的 LF 再读 size+1 字节的 body,天然不串话。大文件命中确认门槛时直接 kill+重生,避免把超大 blob 拽进内存。 Each long-running process is fed requests serially (responses come back on the single stdout in input order); a small state machine parses the header LF then reads size+1 body bytes, so streams never desync. On a large-file gate hit it kills+respawns rather than draining a huge blob into memory.

4跨平台设计:机制通用,命令按平台解析Cross-platform: shared mechanism, per-platform command

协议三平台一致,所以"长驻机制"是共享的;唯一按平台分叉的是"用哪个 git 命令驱动它"。这被收敛进一个纯函数(平台作为入参,便于单测): The protocol is identical everywhere, so the "long-running mechanism" is shared; the only per-platform fork is "which git command drives it". That is collapsed into one pure function (platform injected, so it's unit-testable): 

// electron/main/git-cat-file-batch-platform.ts  (pure, zero deps)
export function resolveBatchGitExecutable(platform, defaultGit) {
  switch (platform) {
    case 'win32':  return defaultGit               // ✅ implemented: system git
    case 'darwin': return resolveDarwinBatchGitExecutable(defaultGit)  // ⬚ placeholder
    default:       return null                     // ⊘ Linux/others: not enabled → per-call
  }
}
🪟 Windows DONE

用解析到的系统 git。EDR 税在这里收益最大,已实测验证。Uses the resolved system git. The EDR tax makes the win largest here — measured & verified.

🍎 macOS PLACEHOLDER

留 resolveDarwinBatchGitExecutable 给将来接入打包的平台特有 git(摆脱系统 git / Xcode CLT 依赖)。过渡期默认用系统 git,机制已生效。Reserves resolveDarwinBatchGitExecutable for a future bundled platform-specific git (no dependency on system git / Xcode CLT). Interim: system git, so the mechanism is already active.

🐧 Linux OFF

原生建进程本就便宜,按范围不启用,直接走 per-call 回退。Native spawns are cheap; intentionally not enabled — uses the per-call fallback.

5健壮性:双层兜底,绝不让 Diff 崩Robustness: two-layer safety, the diff never breaks

readGitFileByRef 把 batch 包在两层保护里: readGitFileByRef wraps the batch in two layers of protection: 

if (gitCatFileBatch.isSupportedPlatform()) {        // 第 1 层:平台门控 / gate
  try {
    const batch = await gitCatFileBatch.readObject(...)  // 快路径 / fast path
    size = batch.sizeBytes; buffer = batch.data; resolvedByBatch = true
  } catch (e) {
    record('cat-file-batch.fallback')               // 第 2 层:运行时降级 / degrade
  }
}
if (!resolvedByBatch) { /* per-call cat-file -s + blob (老路径,慢但正确) */ }
Guarantee 不支持的平台(Linux)根本不进 batch;支持的平台上,进程死了、协议错位了、或任何异常,都被 catch 接住并退回 per-call。最坏结果是"退回老的慢路径",永远不会是"Diff 打不开"。 Unsupported platforms (Linux) never enter the batch; on supported ones, a dead process, a desync, or any error is caught and falls back to per-call. The worst outcome is "revert to the old slow path", never "the diff fails to open".

Correctness 速度不能以正确性为代价。对一个 108KB 的改动文件,batch 读到的内容与 git show :path 逐字节一致(长度相同、SHA-256 相同)。常驻进程读的是同一个 index blob,只是省掉了 fork。 Speed must not cost correctness. For a 108KB changed file, the batch content is byte-for-byte identical to git show :path (same length, same SHA-256). The long-running process reads the same index blob — it just skips the fork.
