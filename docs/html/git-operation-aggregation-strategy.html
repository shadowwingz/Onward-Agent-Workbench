Git 操作聚合优化策略

用更少的、聚合的 git 调用一次拿全所有信息 —— 而不是每个原子操作 spawn 一次 git。针对 EDR 内核 minifilter 让每次 CreateProcess 收数秒税的场景。 Aggregate many operations into ONE git invocation instead of spawning a fresh git per atomic operation — targeting the EDR kernel-minifilter tax of multiple seconds per CreateProcess.

1 · 原则与问题 / Principle & problem

核心原则
Core principle
尽可能把多个操作聚合进一次 git 调用,从中取出所有可能的信息;不要每个原子操作都 spawn 一次 git。因为在 EDR 重税的宿主上,真正的成本是进程创建本身(每次 CreateProcess 被内核 minifilter 扫描),而不是 git 内部计算。 Aggregate as many operations as possible into one git invocation and extract everything from it; do NOT spawn a git per atomic operation. On an EDR-taxed host the real cost is process creation (every CreateProcess is scanned by the kernel minifilter), not git's internal compute.

4 · 聚合方案 / Aggregations

① 仓根/HEAD/branch:一次多查询 + 永久缓存 + 复用 mirrorRepo root / HEAD / branch: one multi-query, cache forever, reuse mirror 同行

现状

getGitRepoMeta rev-parse+getGitRoot rev-parse+checkGitRepository+rev-parse HEAD/branch

2 条解析路径 + 1s TTL → 85 次 spawn。

2 paths + 1s TTL → 85 spawns.

提案

1× rev-parse(多查询)→永久缓存/cwd+branch/HEAD 读 mirror 快照

仓根/git-dir 对 cwd 不可变 → 永久缓存(去掉 1s TTL);两条路径合一;branch/HEAD-oid 直接读 mirror 的 status --porcelain=v2 --branch 快照(已免费)。→ ~3 次(每仓一次)。

root/git-dir immutable per cwd → cache forever; merge the two paths; branch/HEAD-oid read from the mirror's status --porcelain=v2 --branch snapshot (already free). → ~3 (one per repo).

# 同行 (lazygit GetRepoPaths): 一次 rev-parse 多查询拿全仓元数据
git rev-parse --path-format=absolute --show-toplevel --absolute-git-dir \
              --git-common-dir --is-bare-repository --show-superproject-working-tree

② History:一次 git log -p 拿全 N 个 commit 的 diffHistory: one git log -p for N commits' diffs 同行

现状

log(列表)→diff --name-status #1--numstat #1…#N ×2

预热 10 个 commit ≈ 20+ spawn;本机每次 3.7s → ~135s。

10-commit prewarm ≈ 20+ spawns; ~135s here.

提案

1× git log -p --numstat --name-status -z -n N

一次 spawn 流式拿到 N 个 commit 的元数据 + 每个的 patch + stats。20+ → 1。

One spawn streams N commits' metadata + each patch + stats. 20+ → 1.

# 提案:首页 commit 的 diff 一次性拿全(VS Code / git4idea / git 原生都这么做)
git log -p --numstat --name-status -z -n 50 --pretty=format:<fmt> <branchOid>
# 大输出可分两步:先 --numstat(便宜)填列表+文件,-p 按需补 patch

③ Diff stats:一次拿 raw+numstat(已部分聚合)Diff stats: one raw+numstat call

列表的增删行数已是一次 diff --numstat -z 拿全(好)。但 --name-status 与 --numstat 仍是两次。同行(VS Code diffBetweenWithStats)用 一次 diff --raw --numstat -z 同时拿到改动类型 + 增删数 → 2 → 1。

Per-file additions/deletions already come from one diff --numstat -z (good). But --name-status + --numstat are still two spawns. VS Code's diffBetweenWithStats uses one diff --raw --numstat -z for change-type + counts together → 2 → 1.

④ 安装检查 --version:进程内永久缓存--version: cache for process lifetime

git 版本/可用性在进程生命周期内不变 → 缓存一次,后续 0 spawn。

git version/availability is constant for the process lifetime → cache once, 0 spawns after.

关键反例:GitKraken 从 libgit2 迁回 git CLI
Key counter-example: GitKraken migrated FROM libgit2 BACK to the git CLI
libgit2 能彻底消除 spawn,但 GitKraken 因特性对齐/正确性/维护(libgit2 跟进新 git 特性慢、偏 host 而非 client)迁回了 git 可执行文件。结论:不走 libgit2;走"聚合 CLI 调用"这条更稳的路。 libgit2 eliminates spawns, but GitKraken moved back to the git executable for parity/correctness/maintenance (libgit2 trails new git features, is host- not client-focused). Conclusion: do NOT adopt libgit2; aggregate CLI calls instead.
