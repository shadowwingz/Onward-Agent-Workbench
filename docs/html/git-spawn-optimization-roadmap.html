③ 策略 A：FSMonitor（治本，最大杠杆）Strategy A: FSMonitor (the real cure, biggest lever)

同类实测（GitHub 工程博客）：Chromium 39 万文件 git status 17.6s → 0.83s（21×），合成 200 万文件 85s → 0.75s（113×）。在 NTFS 上"全量 refresh 可达数分钟"，FSMonitor 把成本从"总文件数量级"降到"改动文件数量级"。 Peer benchmark (GitHub blog): Chromium 393K files git status 17.6s → 0.83s (21×); synthetic 2M files 85s → 0.75s (113×). On NTFS a full refresh "can take minutes"; FSMonitor shifts the cost from total-file-count to changed-file-count.

A.2 三个坑（对抗验证发现）Three gotchas (found by adversarial verification)

坑 1 -c core.fsmonitor=true 是空操作。临时的 per-spawn -c 标志只是"去问 daemon"，但没有常驻 daemon 时它什么都不做，无法自举 daemon。必须持久化 git config + 确保 daemon 运行（git 首次带 fsmonitor 的 status 会自动拉起 daemon——但那是一次写 + 进程拉起）。-c core.fsmonitor=true is a no-op. A transient per-spawn -c flag only "asks the daemon" — it does nothing without a running daemon and can't bootstrap one. You must persist git config + ensure the daemon runs (git auto-starts it on the first fsmonitor-status — but that's a write + process spawn).

坑 2 与只读契约冲突。本仓刻意用 GIT_OPTIONAL_LOCKS=0 只读跑 git（不写索引、不留锁），避免破坏用户仓库状态。FSMonitor 要写索引 + 在用户仓库起 daemon——这是对一个已确认策略的反转，需你明确接受。Conflicts with the read-only contract. The app deliberately runs git read-only (GIT_OPTIONAL_LOCKS=0; no index writes, no locks) to never corrupt the user's repo. FSMonitor writes the index + starts a daemon in the user's repo — a reversal of a confirmed strategy that needs your explicit acceptance.

坑 3 仅 macOS / Windows。内置 daemon 自 Git 2.37 起仅在 macOS/Windows 提供，Linux 无内置 daemon → 需 process.platform 分支 + Linux 回退（无 daemon 时退化为全量 stat，即现状）。macOS / Windows only. The built-in daemon (Git 2.37+) ships on macOS/Windows only; Linux has no built-in daemon → needs a process.platform branch + a Linux fallback (degrade to full stat = today's behavior).

④ 策略 B：取消被放弃 cwd 的预热Strategy B: cancel abandoned-cwd precompute

B.1 当前 vs 提案Current vs Proposed

协调器有 onCwdAttached（进入仓库即预热）但无 onCwdDetached（离开钩子）。你 13 次/min 疾速切换时，离开的仓库的 precompute burst 仍在低优先级 spawn git（cancelProject 已存在，但只在 LRU 淘汰时调）。The coordinator has onCwdAttached (prewarm on entering) but no onCwdDetached. At 13 switches/min, a left repo's precompute burst keeps spawning git at low priority (cancelProject exists but is only called on LRU eviction).

验证纠正：它不"饿死"当前仓库——每个仓库的 burst 在自己的 ::precompute-burst 车道（并发 1），不会 FIFO 阻塞新仓库。但它确实白烧系统级 EDR minifilter 配额，与你真正落地的前台仓库竞争进程创建吞吐。所以是真浪费，但量级中等（约省 8–12 spawn / 6min 密集会话）。 Verification correction: it does NOT "starve" the current repo — each repo's burst runs in its own ::precompute-burst lane (concurrency 1), no FIFO blocking. But it does burn system-wide EDR minifilter budget, competing with the foreground repo for spawn throughput. Real waste, moderate (~8–12 spawns / 6-min dense session).

B.2 待你决策Decisions needed from you

Q1 立即取消 vs 宽限期？ 立即取消（最省 spawn，但 A→B→A 来回切会丢弃 A 的半成品预热）／ 宽限 2–3s 再取消（避免来回切抖动，推荐）。Q1 Cancel immediately vs grace period? immediate (max spawn savings, but A→B→A thrash discards A's partial prewarm) / 2–3s grace before cancel (avoids back-and-forth thrash, recommended).

Q3 是否同时给"最新选中 cwd"优先级提升？（与取消正交，可叠加，让你落地的仓库最先暖。）Q3 Also boost the latest-selected cwd's priority? (orthogonal to cancel; can stack so the repo you land on warms first.)
