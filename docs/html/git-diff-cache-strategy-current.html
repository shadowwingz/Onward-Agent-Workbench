Git 缓存策略 — 预热架构与各级 Cache 设计

本版围绕一条新总原则重写:凡是要 fork git 的耗时活(Diff 列表 / Diff 内容 / History),都在 cwd 切到 git 仓时由主进程提前全做好;UI 打开 Diff / History 时只「取结果」,不现算。用绘图讲清每一级 cache 的设计、覆盖每个 UI 入口,并把对抗式评审挖出的风险与必做缓解摆清,供你逐项决策。

一句话:"把昂贵的 git 操作提前在后台做好藏进多级缓存,让 UI 只读结果;再用一个'新鲜度权威'在数据真变时精确失效。"

1总原则 + 全景架构图

核心转变:从"进 Diff 时才算"改为"cwd 切到 git 仓时就在后台算好"。UI(Git Diff / Git History)退化为纯读缓存——命中即秒回,只有"预热还没跑完"这一个例外才会看到 loading。

读图要点紫线 = 提案的预热流(cwd→repoRoot→prewarmRepo→三大缓存);灰线 = UI 只读;红线 = 新鲜度权威的失效。三大缓存是并列的三根柱子:列表、内容、History。UI 永远先问缓存,命中即秒回。

2UI 操作入口全景(各入口 → 各级 cache)

你要的"覆盖各种 UI 操作入口":下表把每一个会触发 git 的 UI 动作映射到它读哪级缓存、命中/未命中分别多贵。这就是"UI 只取结果"的落点清单。

UI 入口触发的 git 活读哪级缓存命中未命中(冷)

3被缓存的成本

每一层缓存都对应一种"不缓存就反复付"的昂贵成本。

成本有多贵(实测)为什么贵对应层
创建 git 进程~0.4–3 秒/次(Windows)公司机器 EDR/DLP 内核 minifilter 每次 CreateProcess 同步检查(非 Defender)L4 cat-file batch(常驻免创建)
git status 大树遍历~5–12 秒(kar-qemu 冷)--untracked-files=all 遍历已跟踪树;submodule 递归更贵L5/L6 列表 + L7 结构快照
读文件内容(diff 正文)原 ~1.3s/文件 → 现 1–2ms读 index blob + 工作树;原每次 fork 两个 cat-fileL1/L2 内容 + L4 batch
git log(History)每次 fork 3 个进程(repoMeta + rev-list + log)无任何缓存,每次打开都现算L8 history 缓存(本轮新增)
关键认知"创建进程"成本已被 L4 基本消除(待决策);大头是 git status 与 git log 本身的遍历时间——这正是"提前在后台做好"要藏起来的对象。kar-qemu 一次开 Diff fork ≈ 66–95 个 git 进程(N=20 submodule),每个吃 EDR 税 → 数秒墙钟。

4新鲜度权威:watcher + reconcile + branchOid

所有缓存层都不自己轮询。它们订阅唯一权威 GitStateMirror(后台 worker + 文件监听),由它判断"仓库变了"再广播失效。合并上游后有三个新鲜度信号:

来源 A:@parcel/watcher 快路径 基线

原生 FS 事件 → classifyEventPath 噪声过滤 → 80ms 防抖 → 一次 git status 重算。亚秒级。本轮我加:丢弃原子保存临时文件 *.tmp.<pid>.<hash>本轮:gitignore 目录 → ignore globs

来源 B:reconcile 常驻心跳 上游新增

上游发现 @parcel/watcher 会静默失效(macOS FSEvents 停发事件、不报错不回调,parcel#187:50 分钟 0 fire)。解法:worker 内常驻心跳,与 watcher 并行、不依赖它报错——焦点仓 1s、可见 3s、隐藏不跳;按 repoRoot 去重;带 drift 检测。

来源 C:branchOid(HEAD 变更)提案 · 为 History 而加

当前 MirrorState.changeFingerprint = sha1(rawStatus + Σ mtimeNs:size:mode),只反映工作树/index 的变化,不含 HEAD 的 commit oid。这对 Diff 够用,但对 History 不够:

为什么 History 必须加 branchOid(对抗评审 HIGH)git commit --amend -m '改消息'(树没变)→ changeFingerprint 不翻转 → 不发 mirror-update → 若 History 缓存只按 repoRoot::limit::skip 键,会把旧提交消息陈旧缓存 30 分钟。必做缓解:给 MirrorState 增 branchOid(git rev-parse HEAD),History 缓存键改 repoRoot::branchOid::limit::skip,branchOid 变即失效。
FS 事件(watcher) · reconcile 心跳(1s/3s) · branchOid 变更(HEAD)
↓ 过滤 + 防抖 + 指纹/oid 判定
worker:git status --porcelain=v2 重算 → 算指纹 + 读 HEAD oid
↓ 指纹翻转 或 oid 变 才发
emit mirror-update(cwd) → gitDiffCacheInvalidator
↓ 扇出
列表 / 内容(scoped) / 快照 / History(按 branchOid) 各自失效

6预热触发:现状 vs 提案(Option A)

"提前做好"成立的前提是预热在正确时机触发、且只触发一次、且不饿死前台。当前触发链绕道 renderer;提案把它迁到主进程 repoRoot 解析处。

现状(renderer 触发)
主进程 mirror 算出 repoRoot
↓ 发 IPC GIT_TERMINAL_INFO
renderer onTerminalInfo(TerminalGrid)
↓ leading-edge / 2s 防抖
renderer 回调 warmDiffCache(repoRoot) IPC
↓ 回到主进程
才开始暖 · 多绕一圈 IPC · 依赖 renderer 挂载
提案(主进程 repoRoot 即触发)
主进程 attachMirror 解析出 repoRoot
↓ 同进程直调(无 IPC 往返)
prewarmRepo(repoRoot) · leading-edge 一次
↓ 低优先级 · fire-and-forget
暖 列表 + 内容(+ History 首页)
↓ 退役 renderer 触发,避免双触发
更早启动 · 不依赖 renderer · 单一权威

提案的三条硬约束(对抗评审钉出来的)

  • 只在 attachMirror leading-edge 触发,不在 handleMirrorUpdate §11:后者每次 git 状态变都触发,会与 reconcile 心跳并发风暴(两个 getDiff(full) 同仓并行 = 140 进程)。
  • per-repoRoot 去重:多终端在同一仓 → 只 prewarm 一次(靠 lastPrewarmedCwds Set 或 git-runtime in-flight dedup,二选一,见 §12)。
  • 退役 renderer 的 warmDiffCache:否则主进程 + renderer 双触发,竞态下 2× ~69 进程烧 EDR 配额。可加临时开关 ONWARD_USE_MAIN_SIDE_PREWARM 过渡。

车道隔离:预热绝不能堵住前台点击 §11 HIGH

低优先级"只防排队、不防正在跑":一个 in-flight 的低优先 spawn 跑着时,高优先点击仍要等它跑完。所以必须分车道(repoKey),让前台点击走独立车道:

${cwd}前台:用户点文件 / 进 Diff —— 高优先,独占车道priority: high
${cwd}::diff-precompute预热列表:warmDiffCache 两段 —— 低优先priority: low
${cwd}::precompute-burst预热内容:scheduler 并发取文件 —— 低优先,与前台点击分开priority: low
必做缓解当前 scheduler 的内容预取与前台点击可能共用 ::precompute 车道,会让点击排在 6 个并发预取后等 18–29s。必须把 scheduler burst 拆到独立车道(或前台点击单独 ::precompute-foreground)。仍残留:EDR 下单个 in-flight spawn 无法中途打断,首点最坏仍等 0.4–3s(需 spawn 取消才能根治,本轮先文档化)。

7读取路径:Diff 点文件 / History 看 commit

Diff:点一个文件的完整旅程

① L1 渲染端内存命中? → renderer-memory HIT(不过 IPC,最快)
↓ miss / 被标 stale / force
② L2 主内容缓存命中?(get 不看 generation) → main-content HIT
↓ miss
③ L3 去重 → 取对象:unstaged=1 次 index blob + 1 次工作树读;staged=2 次对象读
↓ 经 L4 cat-file batch 管道(win32 无新建进程)
回填 L2(带 staleToken)+ L1 → 显示。MISS(冷读),但极快

History:看一个 commit 的旅程 本轮新增 L8

① 开 History → 问 L8(repoRoot::branchOid::50::0) → 命中秒回首页
↓ miss(branchOid 变了 / 没预热到)
② worker git log -n 50 + rev-list --count → 回填 L8
↓ 用户点某 commit / 翻页
③ git diff A..B / 翻页 skip=50 —— 本轮暂按需(不预热,见 §2/§12)

"miss" 的几种原因(调试面板可见)

missReason含义
first-load该 project 桶还不存在(从没读过这个仓)
entry-not-warmed桶在、但这个文件还没被预热到
precompute-pending预热正在跑、还没轮到它("首点 miss"典型)
invalidated-*刚被 mirror/force 失效清掉

8失效:一次变化如何清缓存

一次 mirror-update 经 gitDiffCacheInvalidator.invalidate(cwd, reason) 扇出到监听器:

监听器清哪些层粒度
git-utilsL5 请求 + L6 单仓 + L7 快照整体(聚合层)
content-cache-wiringL2 内容本轮:reason='mirror' 逐文件 re-validate;其余整桶
ipc-handlersworker 副本 + 通知渲染端 + 重排预热基线
history-cacheL8 History提案:仅 branchOid 变更时清(非每次 dirty)
为什么列表整体清、内容可逐文件清、History 按 oid 清列表/分支是相互关联的聚合状态 → 整仓失效(同行做法,细粒度有漏清风险);单文件内容可分解 → 只丢真变的(§11① scoped);History 只随 HEAD 移动而变 → 按 branchOid 清,工作区改文件不该动它。

scoped 失效的具体机制 本轮

// mirror 失效时:不整桶清，而是逐条 re-stat、只逐出真变的
revalidateProject(project, (key, staleToken) => {
  const { changeType, filename } = parseCacheKey(key)
  if (!isWorktreeBacked(changeType)) return true   // staged/索引侧 → 保守逐出
  if (staleToken === undefined) return true     // 无 token → 保守逐出
  const now = statToken(project + filename)        // 重新 stat 工作树文件
  return now !== staleToken                       // 变了 → 逐出；没变 → 保留
})
// >1000 条 → 退回整桶清

9核心不变量:generation / 指纹 / branchOid

① generation(内容缓存代际)

  • get() 不看 generation → 已缓存条目永远能读到。
  • fetch 开始时记 generation;完成时只在仍是那代才 put → 防"fetch 途中失效"存陈旧。

失效逐出会 bump generation。这是 scoped 失效安全的根:逐出变化文件 + bump → 任何"变化前开始、变化后完成"的 fetch 被丢弃;未变文件不受影响。

② 变更指纹(Diff)

fingerprint = sha1( rawStatus + Σ(changedFile, mtimeNs:size:mode) )   // 排除 ctime（NTFS churn 防护）

③ branchOid(History)提案新增

branchOid = git rev-parse HEAD   // 进 MirrorState；History 缓存键含它，oid 变即失效
historyKey = `${repoRoot}::${branchOid}::${limit}::${skip}`
三者口径一致内容层 staleToken 用同一个 mtimeNs:size:mode(逐文件),与指纹判定口径一致;branchOid 是正交的第三维,专管 HEAD 移动——三条线互不串扰。

产品它怎么做对我们的含义

JetBrainsVCS Log 持久化增量索引:仅首建全量(IntelliJ 仓 ~15 分钟),之后只索引新 commit;空闲时跑、批量、可取消、可关。"全量预热 history"代价不可接受;我们取"首页 + 按需翻页",更轻。
git 本体fsmonitor 常驻守护 + untracked cache(目录 mtime);token 含 PID+syncID,不确定就退回全扫,绝不返陈旧;commit-graph(generation number)让 git log 快 ~650×。失效铁律"不确定→重算";可考虑保证 commit-graph 常暖作为 log 的补充加速(但 EDR 下仍需 in-app 缓存首页)。
GitKraken"开仓快 5×"= 后台跑 git maintenance 暖 commit-graph;给图设"Max Commits"上限。预热 = 优化底层 git 结构 + 给图封顶,而非内存里堆大图。
同行共识 = 我们的方向status/列表预热 · history 只首页 + 分页 · 巨仓封顶 · 按 repo 去重合并 · 不确定即重算。唯一被反复点名的硬约束:巨仓的内存/成本封顶(对应 §11 的 maxProjects 与 per-burst 上限)。

11风险与必做缓解(对抗式评审)

4 个对抗 lens(内存/新鲜度/并发/跨平台)对提案逐条找漏。方向成立,下列全部 HIGH/MED 缓解已在 §12 采纳(每项均经代码核验)。保留风险描述作为"为什么这么改"的依据。

HIGH前台点击被 in-flight 预热饿死(并发)
场景预热与前台点击共用 repoKey 车道时,高优先点击排在 in-flight 的低优先 getDiff/内容预取后,EDR 下可等 10–30s,违反"不得饿死前台"。 必做缓解车道隔离(见 §6):前台 ${cwd} / 预热列表 ::diff-precompute / 内容 burst ::precompute-burst 三独立车道;scheduler burst 不得与前台点击同车道。
HIGHHistory 缓存对 amend/checkout 陈旧 30 分钟(新鲜度)
场景git commit --amend(树没变)→ changeFingerprint 不翻转 → 不发失效;若 History 键不含 branchOid,旧消息陈旧 30 分钟。 必做缓解MirrorState 增 branchOid;History 键 = repoRoot::branchOid::limit::skip;branchOid 变即失效(见 §4 来源 C)。
HIGHmaxProjects=8 < kar-qemu 20 submodule → 淘汰抖动(内存)
场景内容缓存只留 8 个 project 桶;kar-qemu 20 submodule 全量预热到第 9 个就把父仓挤掉。"submodule 首点命中"对 >7 submodule 的仓不成立。 必做缓解把 maxProjects 由 8 提到 16–24(或加 ONWARD_DIFF_CACHE_MAX_PROJECTS 环境变量);否则 ② "含 submodule 全量预热"的收益被淘汰抵消。
HIGH单批文件数无界 → 撑爆桶预算(内存)
场景maxCandidatesPerBurst:1000 是安全天花板;500+ 文件 × ~100KB 跨多 submodule 会超 100MB/桶,触发"最小先淘汰",留下大文件反复重取。 必做缓解每批硬上限 ~100 文件,超出分多批;或缓存 >70% 满时暂停预热,给前台大文件让路。
MEDwatcher 静默失效时列表不失效(新鲜度)
场景watcher 崩溃/超时若未把 watcher-error 接到 gitDiffCacheInvalidator.notifyWatcherError,列表缓存可陈旧到 30min TTL。 缓解核验并接好 watcher-error → 失效;reconcile 心跳的 drift 检测是部分安全网(它发现变化会重新失效),但应显式补齐 watcher-error 接线。
MED内容缓存 stat 碰撞返回陈旧(新鲜度)
场景外部改文件但 watcher 漏事件,且 mtime:size 恰好不变(保存后立刻打开的 TOCTOU)→ re-stat 误判"未变"→ 返回陈旧正文。 缓解命中时若 staleToken 年龄 > 2s 则强制重取(宽限期);或命中即同步 re-stat。
MEDcwd 快速切换不取消在飞预热(并发/跨平台)
场景cd a; cd b:repo-a 的 ~69 进程预热已入队,用户已在 repo-b,预热仍跑完,白烧 EDR 配额(非正确性 bug)。 缓解"nice-to-have":detachMirror 时用 AbortController 取消;或文档化为已知次要浪费(低优先 + 30min 缓存使其无害)。评审 A5 倾向不取消(低优先自然无害);A1/并发倾向取消。见 §12 决策。
MEDmacOS cwd 规范化竞态(跨平台)
场景macOS /var vs /private/var 符号链接,若 prewarm 与 renderer/worker 规范化口径不一,缓存键不匹配 → 白预热。 缓解规范化只在 subscribe 一处做,prewarmRepo 复用已规范化的 cwd,不重复 realpath;加 trace 检测口径分叉。
评审总判定内存组 = SAFE-with-mitigations(3 项 HIGH 需改);新鲜度组 = 需先修 branchOid/stat/watcher-error 三项;并发组 = 需车道隔离 + attachMirror-only + 退役 renderer 触发;跨平台组 = SAFE-with-mitigations(取消 + 规范化)。结论:方向对,但上面 4 个 HIGH 是动代码前要在 §12 拍板纳入的前置条件。
