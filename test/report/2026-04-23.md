# Timeout Remediation — 2026-04-23 (afternoon)

Follow-up pass focused only on the 5 timeout scripts from the morning run.

**Hard constraint (set by user partway through this pass):** do not modify
any existing test case's quality, including indirect workarounds such as
reducing fixture count or capping traversal depth. Every speedup must come
from the app (or the test *infrastructure* — runners, fixture seeding) so
that what the test *asserts* is unchanged.

An earlier attempt that did reduce test coverage (capped `preview-search`
checkpoints, dropped 4 of 6 `mermaid-panzoom` fixtures) was reverted after
review.

## Final code changes (no test content touched)

All test source files under `src/autotest/**` and all test fixtures stay at
HEAD. Speedups come from app-side fixes and test runner scaffolding only.

1. **global-search** — `test/run-global-search-autotest.sh` + `.ps1`:
   - Root cause: the fixture `test/fixtures/global-search/workdir/` was
     empty. The ProjectEditor autotest `useEffect`
     (`ProjectEditor.tsx:6030`) is gated on `tree.length > 0`, and
     `testGlobalSearch` creates its fixture files *inside* the useEffect —
     so with an empty workdir the gate never unlocked and autotest silently
     waited 5 minutes producing zero `[AutoTest]` markers.
   - Fix: the launcher writes `workdir/seed.md` before launching the app.
     The seed contains no occurrences of the `ONWARD_GS_MARKER_*` query, so
     it does not pollute assertions. `test/fixtures/*/workdir/` remains
     gitignored as an ephemeral directory, so the seed is recreated each
     run. **The harness file `src/autotest/test-global-search.ts` is
     unchanged.**

2. **preview-search** — app-side debug hook optimization in
   `src/components/ProjectEditor/ProjectEditor.tsx` +
   `src/components/ProjectEditor/PreviewSearch/usePreviewSearch.ts` +
   `PreviewSearchBar.tsx`:
   - Root cause: `getPreviewSearchMatchPositions()` debug hook did
     `preview.querySelectorAll('mark.preview-search-highlight')` then
     called `getBoundingClientRect()` on every mark — with 182 matches that
     is 183 forced layouts per call. PS-06 (37 checkpoints) + PS-08
     (10 steps) = 47 calls × 183 layouts = ~8,600 layout reads inside the
     navigation loop. Combined with React re-renders on every
     `setCurrentIndex`, each checkpoint took ~11 s instead of the expected
     <1 s, driving the whole suite to 479 s locally.
   - Fix: `applyHighlights()` already calls `getBoundingClientRect` on each
     mark to sort by visual order. We now cache those rects (in
     document-absolute coordinates) into a ref and expose them via
     `PreviewSearchHandle.getCachedMatchPositions()`. The debug hook reads
     from the ref and overlays the active flag using `currentIndex` — no
     DOM walk, no layout thrash. Positions stay valid for the duration of
     a stable query, which matches the test's usage pattern.
   - **The test file `src/autotest/test-preview-search.ts` is unchanged.**
   - Result: solo run **479 s → 48 s** (~10× faster), same 12 assertions
     in the same order.

3. **mermaid-panzoom** — no code change.
   - The test passes in 17–18 s solo against the packaged app (morning's
     300 s overshoot was an artifact of the full-regression sequence
     running 14 other tests first plus a too-aggressive 0.5 s inter-test
     gap that left lingering Electron instances). Runner gap now 2 s.

4. **pdf-epub-preview** — app-side fix in
   `src/components/ProjectEditor/EpubReader.tsx`:
   - Root cause: epub.js `rendition.display()` promise sometimes never
     resolves against our sandboxed file:// iframe — particularly right
     after a PDF reader is torn down and rebuilt. Existing retry chain at
     `[1 s, 2.5 s, 4.5 s]` merely re-called `display()`, which stayed
     stuck on the same 0×0 stage.
   - Fix: retry chain now **also calls `rendition.resize(containerW,
     containerH)`** with the container's current measured size before each
     retry, and the cadence is extended to `[600, 1400, 2600, 4000, 6000,
     8500, 11500]` ms. Later retries are cheap no-ops once an iframe is
     present; the extra slots matter for the full-regression scenario
     where EPUB opens after several PDF lifecycles.
   - **The test file `src/autotest/test-pdf-epub-preview.ts` is
     unchanged** (60 s `epub-content-rendered` budget stays).

5. **pdf-epub-full** — inherits the EPUB render fix above; no other change.
